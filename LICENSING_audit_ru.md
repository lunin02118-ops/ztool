# Аудит системы лицензирования ZTool и оценка сложности смены сервера лицензий

Версия: детализированная. Документ основан на статическом анализе распакованной
(деобфусцированной) сборки `ZTool.exe` и вспомогательной сборки `ZTool_rsa.dll`
(уровень IL/CIL). Это **аналитический отчёт**: генератор ключей, «кейген» или
совместимый сервер не создавались и в объём работы не входили.

> **Правовое замечание.** ZTool — сторонний коммерческий продукт. Модификация механизма
> лицензирования, перевыпуск ключей или перенос на собственный сервер допустимы только при
> наличии прав/разрешения правообладателя. Ниже — оценка технической сложности и описание
> архитектуры, а не руководство по обходу защиты.

---

## 1. Методология и объект анализа

- Сборка распространяется упакованной протектором **ConfuserEx (Compressor)**: строковые
  литералы зашифрованы, тела методов защищены. Для анализа использовалась снятая из памяти
  и пересобранная сборка (см. отдельный отчёт по русификации `AUDIT_ru.md`).
- Анализ проводился по IL: перечисление типов/методов, литералов, ссылок на классы
  `System.Net.*`, `System.Security.Cryptography.*`, `System.Management` (WMI),
  `Microsoft.Win32.Registry`, а также по дизассемблированию ключевых методов.
- Криптоконстанты, адрес сервера и парольные фразы восстановлены статически и подтверждены
  расчётом (например, адрес сервера вычислен из зашитого шифртекста встроенным открытым ключом).

## 3. Машинный отпечаток (привязка к «железу»)

Сборка идентификатора ПК (`ZTool.SR`):

1. **Диск** — WMI `SELECT … FROM Win32_PhysicalMedia` → свойство `SerialNumber`
   (`GetDiskSN2_1`; есть варианты `GetDiskSN1/1_1/2`).
2. **Материнская плата** — WMI `Win32_BaseBoard` → `SerialNumber` (`GetBoardId`).
3. **UUID системы** — WMI `Win32_ComputerSystemProduct` → `UUID`, валидируется regex
   `^[A-F0-9]{8}(-[A-F0-9]{4}){3}-[A-F0-9]{12}$` (`GetUUID`).
4. **MAC** — описание/адрес сетевого адаптера (`MacAddress.GetNic`).
5. **Нативный резерв** — серийный номер диска через `DeviceIoControl` в `ZToolARM.dll`
   (вызывается с маркером-строкой `"冰雨。。。"`; имя библиотеки `ZToolARM.dll`, при отсутствии —
   сообщение `"ZToolARM.dll"丢失`).

Полученные значения склеиваются через разделитель `|`, после чего:

```
машинный_код = RSAHelper.EncryptString(склейка, ОТКРЫТЫЙ_КЛЮЧ)   // SR.GetMNum, m^e mod n
```

То есть «машинный код», который пользователь отправляет автору/серверу, — это RSA-шифртекст
аппаратных данных открытым ключом (расшифровать может только держатель закрытого ключа).

## 4. Криптопримитивы

### 4.1 RSA-1024 (собственная реализация, `ZTool_rsa.dll`)
- `DWKEYSIZE = 1024`, экспонента `e = 65537`.
- Ключ передаётся строкой `ComponentKey` = `Base64( [len(exp)] · exp · modulus )`.
  `ResolveKey` разбирает её обратно в `(exp, modulus)`.
- **`EncryptString(text, key)`** — поблочно (по 128 символов), `m^e mod n` через `BigInteger.modPow`,
  блоки кодируются `ToHexString()` и соединяются символом `@`.
- **`DecryptString(text, key)`** — также `c^e mod n` с той же (открытой) экспонентой; фактически это
  **проверка/разворот RSA-подписи**: данные, «подписанные» закрытым ключом сервера (`m^d mod n`),
  корректно разворачиваются открытым ключом клиента.
- **Встроенный ключ — только ОТКРЫТЫЙ.** Декодирование строки
  `AwEAAZ5qj1upl…WyshiR` даёт: `len(exp)=3`, `exp=0x010001 (65537)`, `modulus=128 байт (1024 бита)`,
  **закрытой экспоненты `D` в сборке нет**. Закрытый ключ находится только у автора/сервера.

> Следствие: клиент способен (а) зашифровать данные ДЛЯ сервера и (б) проверить подпись лицензии
> ОТ сервера, но **не способен сам выпустить валидную лицензию** — для этого нужен закрытый ключ.

### 4.2 AES (`ZTool.SecurityCenter`)
- `Aes.Create("AES")`; `key = IV = MD5( Encoding.Unicode.GetBytes(парольная_фраза) )` (16 байт);
  `BlockSize = 128`, `Mode = CBC`, `Padding = PKCS7`; результат — Base64 (`EncriptStr`),
  обратное — `DecriptStr`.
- Парольные фразы зашиты строками (китайские названия песен), используются для разных полей:
  `冰雨。。。`, `忘情水。。。`, `今天。。。`, `来生缘。。。`.

### 4.3 DES (офлайн-путь)
- `FrmRg.Decryptstr` (`DESCryptoServiceProvider.CreateDecryptor`) и `Program.EncryptDES`
  (`CreateEncryptor` + `CryptoStream` + Base64) — применяются к офлайн-файлу/коду авторизации.

## 5. Хранение и валидация лицензии (реестр)

Лицензия хранится в **HKEY_LOCAL_MACHINE** в нескольких замаскированных ветках с
перекрёстной проверкой (анти-подмена/анти-откат):

| Ветка | Значение |
|---|---|
| `HKLM\Software\SolURxxCfNU\C4eHN4fjikBan` | `information` |
| `HKLM\Software\SolURxxCfNU\HTwk2RCBDL` | `information` |
| `HKLM\SOFTWARE\Microsoft\MzORu8qE4HhZ\Jlj4aG8uZBvW` | `F2S6qCdziIAm` |
| `HKLM\SOFTWARE\Microsoft\MzORu8qE4HhZ\98CqyvBZcg` | `information` |

Проверка (`SR.IsReg1` / `SR.IsReg2`):
1. Прочитать значения из веток реестра.
2. Снять AES-слой: `SecurityCenter.DecriptStr(value, парольная_фраза)`.
3. Снять RSA-слой / проверить подпись: `RSAHelper.DecryptString(…, ОТКРЫТЫЙ_КЛЮЧ)`.
4. Разобрать поля (разделители `|` и `\n`): машинный код, время регистрации (`get_rgtime`,
   пишется как `DateTime.Now.ToString()`), сведения о регистрации (`get_rginfo`).
5. Сверить машинный код с текущим ПК (через `Operators.CompareString`) и проверить срок.
   Несколько копий в разных ветках сверяются между собой — модификация одной ломает валидацию.

## 6. Онлайн-активация (протокол `ZTool.TCPClient`)

### 6.1 Адрес сервера
- Поля `ip_address` и `ip_port` зашиты в **зашифрованном** виде и в `Connect()` разворачиваются
  встроенным открытым ключом, затем `IPAddress.Parse` + порт в `Socket.BeginConnect`.
- **Восстановленный адрес: `120.78.215.80 : 58000`** (диапазон Alibaba Cloud, КНР). Получен расчётом
  `c^e mod n` по зашитому шифртексту и встроенному открытому ключу.
- Таймаут соединения контролируется (`ManualResetEvent`); ошибки: `网络异常`, `连接服务器超时`,
  `连接服务器出错`.

### 6.2 Формат кадра
`sendstring(type, data)` → тело шифруется AES, затем `getsendbuffer` собирает кадр:

```
payload = SecurityCenter.EncriptStr( data, ToString((int)type) )   // ключ AES = номер типа сообщения как строка, напр. "130"
body    = UTF8( payload.Trim() )
frame   = int_to_Byte(type, 10)  ‖  int_to_Byte(len(body), 10)  ‖  body
```

- `int_to_Byte(v, 10)` укладывает `v` в 10-байтовое поле как little-endian (значимы младшие 4 байта,
  остальные нулевые). То есть кадр = `[тип:10][длина_тела:10][AES-Base64 тело]`.
  Поля `H_length = L_length = 10` в классе подтверждают длину заголовков.
- Особенность: **ключ AES для тела — это строковое представление кода сообщения** (`Sendtype`).

### 6.3 Коды сообщений (`Sendtype`)
| Имя | Значение | Назначение |
|---|---|---|
| `Apply_register` | 128 | заявка на регистрацию (активацию) |
| `Apply_Remove` | 129 | заявка на снятие/перенос лицензии |
| `register` | 130 | регистрация |
| `verify_register` | 131 | проверка регистрации |
| `verify_Remove` | 132 | проверка снятия/переноса |

### 6.4 Ответы сервера (`SocketRecive`)
Расшифровываются и сопоставляются с текстами статусов:
`注册成功` (успех), `注册失败` (отказ), `无效注册码` (неверный код),
`注册信息错误` (ошибка данных), `注册申请失败` (отказ заявки),
`注册码已被其它电脑使用` (код занят другим ПК), `注册码已过期` (код истёк),
`授权电脑数量已达上限` (достигнут лимит устройств), `密码错误` (неверный пароль),
`注册信息保存错误` (ошибка сохранения), а также для переноса:
`转出授权成功/失败` (перенос успешен/неуспешен), `无需转出` (перенос не требуется),
`此电脑没有转出权限` / `此注册码没有转出权限` (нет прав на перенос).

При успехе сервер возвращает подписанную регистрацию, которая раскладывается в реестр (раздел 5).

## 7. Офлайн-активация и перенос лицензии (`ZTool.FrmRg`)

- Для ПК без интернета: машинный код передаётся автору (QR-код на форме, почта `mail@z-tool.cn`,
  страница покупки на `item.taobao.com`). Автор возвращает файл/код авторизации, который клиент
  расшифровывает через DES (`Decryptstr`) и сохраняет в реестр.
- Форма поддерживает **перенос авторизации** («转出授权») с опциональным «паролем защиты
  авторизации» (`授权保护密码` — задаётся при активации, очищается при переносе) — это floating-механика
  возврата лицензии на сервер для активации на другом ПК.

---

## 8. Триал, старт и обновления

- `FrmRverify` на старте проверяет лицензию; без неё — режим триала (`试用`).
- `Frmmain.TestTime_Tick`: `试用中...剩余 N 秒` → по истечении `试用时间到！软件将在10秒后自动关闭！`.
- `ZTool.CheckUpdate`: `WebClient`/`HttpWebRequest`, MD5-проверка, источник `www.z-tool.cn` — **отдельный
  от лицензирования** канал, переносится тривиально.

## 9. Защита от вскрытия

- Упаковка **ConfuserEx Compressor** + self-check `code::HasShell` + перекрёстная проверка реестра.
- В рамках задачи русификации распаковка и пересборка сборки уже отлажены, поэтому статическая правка
  констант/ключей технически доступна (но ломает оригинальную целостность и подпись).

## 10. Оценка сложности смены сервера лицензирования

Главный принцип: **сменить адрес — легко, но заставить клиент ПРИНИМАТЬ лицензии с чужого сервера —
сложно**, потому что доверие построено на асимметрии — клиент проверяет подпись встроенным **открытым**
ключом, а подписывает лицензию **закрытым** ключом только оригинальный сервер.

### Сценарий A. Есть доступ к оригинальному серверу / закрытому ключу автора
**Сложность: низкая (часы).**
- Переэшифровать новый `IP`/`порт` тем же открытым ключом (`m^e mod n` доступна кому угодно) и заменить
  константы `TCPClient.ip_address`/`ip_port`, либо пропатчить `Connect` на открытый адрес.
- Пересобрать сборку (механизм отлажен).
- Реалистично только при сотрудничестве с правообладателем.

### Сценарий B. Свой сервер, оригинального закрытого ключа НЕТ
**Сложность: средне-высокая, оценочно несколько дней — ~1–2 недели; технически выполнимо, но хрупко
и требует модификации клиента.** Необходимо:

1. **Перевыпуск ключей и переподпись клиента** — заменить встроенный **открытый** RSA-ключ на свой
   (в `SR`, `TCPClient.Connect`, ресурсы `PKEY`/`G_SN`) и пересобрать; ваш сервер подписывает закрытым
   ключом, клиент доверяет.
2. **Замена/патч адреса** сервера (`ip_address`/`ip_port`).
3. **Совместимый сервер-сокет** — воспроизвести протокол: кадр `[тип:10][длина:10][AES-Base64 тело]`,
   ключ AES = `ToString(код)`, коды `Sendtype` 128–132, тексты-статусы ответов, последовательность
   заявка→регистрация→проверка.
4. **Серверная выдача лицензии** — подписывать закрытым ключом блоб в точном формате `SR.IsReg1/IsReg2`
   (после AES-слоя и раскладки по 4 веткам реестра), с привязкой к машинному коду и сроку.
5. **Алгоритм машинного кода** — повторить склейку источников и RSA-обёртку (восстановимо из клиента).
6. **Все потоки** — онлайн, офлайн (DES), перенос/возврат (`转出授权`), запись во все ветки реестра и
   прохождение перекрёстных проверок.
7. **Тестирование** — устойчивость к anti-tamper, отсутствие регрессий в надстройке SolidWorks.

### Сценарий C. «Бесшовно», без модификации клиента и без закрытого ключа
**Невозможно.** Не имея закрытого ключа, выпустить лицензию, принимаемую немодифицированным клиентом,
нельзя (если только в реализации RSA/протокола нет отдельной уязвимости — её целенаправленный поиск в
объём аудита не входил; обращаем внимание, что padding в RSA фактически отсутствует — «учебная» схема,
что является слабостью, но её эксплуатация выходит за рамки данного отчёта).

### 10.1 Оценка трудозатрат (ориентир)
| Работа | Сценарий A | Сценарий B |
|---|---|---|
| Замена адреса сервера | 0.5–1 ч | 0.5–1 ч |
| Перевыпуск ключей + переподпись клиента | — | 0.5–1 день |
| Реализация сокет-протокола (сервер) | — | 2–4 дня |
| Серверная выдача/подпись лицензий в нужном формате | — | 2–4 дня |
| Воспроизведение машинного кода + привязка | — | 0.5–1 день |
| Офлайн/перенос лицензии | — | 1–2 дня |
| Тестирование (онлайн/офлайн/SolidWorks/anti-tamper) | 0.5 дня | 2–3 дня |
| **Итого** | **≈ 1 день** | **≈ 1–2 недели** |

### 10.2 Реестр рисков
- **Юридический**: модификация чужой защиты/перевыпуск ключей без разрешения правообладателя.
- **Хрупкость**: перекрёстные проверки реестра, anti-tamper ConfuserEx и проверка `HasShell` — любое
  расхождение формата ломает валидацию «молча».
- **Совместимость**: `ZTool.exe` тесно связан с надстройкой `ZTool.dll` (SolidWorks); пересобранный
  клиент нужно регрессионно проверять в реальном SolidWorks.
- **Безопасность сервера**: вы берёте на себя выпуск/хранение закрытого ключа и доступность сервиса
  (текущий — единая точка `120.78.215.80:58000`).
- **Обновления**: автообновление (`CheckUpdate`) может «перетереть» пропатченный клиент сборкой с
  оригинального сайта — канал обновлений тоже нужно переориентировать или отключить.

## 11. Рекомендации

- Для **легитимной** миграции сервера оптимально запросить у правообладателя: закрытый ключ и/или
  готовую сборку сервера со спецификацией протокола. Это убирает реверс-инжиниринг, хрупкость и
  юридические риски.
- Если цель — независимость от внешнего сервера, корректнее договориться о передаче исходного кода
  (тогда смена сервера/ключей — штатная конфигурация, а не патч бинарника).
