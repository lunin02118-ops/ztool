# План миграции лицензирования ZTool на собственный сервер (Сценарий B)

Детальный пошаговый инженерный план: **свой сервер без оригинального закрытого ключа** →
перевыпуск ключей и пересборка клиента + совместимый сокет‑протокол + серверная выдача
лицензий в точном формате (реестр/AES/RSA) + воспроизведение алгоритма машинного кода.

Базируется на отчёте `LICENSING_audit_ru.md`. Оценка: **средне‑высокая, ~1–2 недели**;
технически выполнимо (распаковка/пересборка отлажены при русификации), но **рискованно и хрупко**.

> **Правовое условие (обязательно).** Работы допустимы только при наличии прав на продукт ZTool
> и/или разрешения правообладателя. План описывает миграцию **вашей** инфраструктуры лицензий,
> а не обход чужой защиты. Перед стартом зафиксируйте правовое основание.

---

## 0. Целевая архитектура и принцип

Сейчас доверие держится на асимметрии: клиент содержит **только открытый** ключ RSA‑1024 и
проверяет им подпись лицензии; подписывает лицензию **закрытым** ключом только сервер автора.
Поскольку оригинального закрытого ключа у нас нет, мы **перевыпускаем собственную пару ключей**
и **встраиваем свой открытый ключ в клиент** (пересобираем его), а наш сервер подписывает своим
закрытым ключом. Форматы хранения (реестр, AES‑слой, разбиение полей) сохраняем прежними, чтобы не
переписывать клиентскую логику валидации.

```
[ Пересобранный клиент с НАШИМ открытым ключом ]
        │  TCP 58000 (или новый адрес), кадр [тип:10][len:10][AES-Base64]
        ▼
[ Наш сервер активации ]  --(подпись НАШИМ закрытым ключом)-->  лицензионный блоб
        │
        └── БД: коды, привязки устройств, лимиты, сроки, переносы
```

Что НЕ меняется (переиспользуем как есть): алгоритм машинного кода в клиенте, AES‑слой
`SecurityCenter`, раскладка по 4 веткам реестра, кадрирование сокета, набор кодов `Sendtype`.
Что меняется: пара ключей RSA (наша), адрес сервера, и **серверная сторона целиком**.

## Фаза 1. Полная спецификация форматов (1–2 дня)

Цель — закрыть все «белые пятна» до написания кода. Зафиксировать в спецификации (`SPEC.md`):

1. **Машинный код** (`SR.GetMNum`):
   - источник: `UUID|disk|board` (UUID из `Win32_ComputerSystemProduct`, диск `GetDiskSN1_1`→fallback
     `GetDiskSN2_1`/`Win32_PhysicalMedia`, плата `Win32_BaseBoard`), trailing `|` обрезается;
   - опционально `+ "\r\n" + version` (через `code.getver("今天。。。")`);
   - берётся `Mid(s,1,117)` (первые 117 символов);
   - затем `RSAHelper.EncryptString(s, ОТКРЫТЫЙ_КЛЮЧ)` (m^e mod n, блок 128, hex, разделитель `@`),
     либо `code.GD51(s)` без RSA — зафиксировать, какой путь используется при онлайн‑заявке.
2. **Запрос активации** (`FrmRg.createinfo`): код регистрации = `L1-L2-L3-L4-L5` (5 полей через `-`),
   опц. пароль (8–20, буквы+цифры). В запрос входят `RSA(код)`, `RSA(пароль)` и машинный код,
   объединённые разделителем `\` и зашифрованные `RSA(...)`. Точно выписать порядок полей.
3. **Кадр сокета**: `[int_to_Byte(type,10)] ‖ [int_to_Byte(len(body),10)] ‖ body`,
   `int_to_Byte` = little‑endian в 10 байт; `body = UTF8(AES_Base64.Trim())`;
   ключ AES тела = `ToString((int)type)` (например `"128"`).
4. **Коды сообщений** (`Sendtype`): 128 `Apply_register`, 129 `Apply_Remove`, 130 `register`,
   131 `verify_register`, 132 `verify_Remove`. Сопоставить каждый код с ожидаемым телом запроса/ответа.
5. **Ответы сервера** (`SocketRecive`): строки‑статусы (`注册成功`, `无效注册码`, `注册码已过期`,
   `授权电脑数量已达上限`, `转出授权成功` и т.д.) + полезная нагрузка лицензии при успехе.
   Определить точный формат успешного ответа (что именно кладётся в реестр).
6. **Лицензионный блоб в реестре**: поля разделяются `\n`; каждое поле проходит
   `RSAHelper.DecryptString(…)` (разворот подписи) и/или `SecurityCenter.DecriptStr(…, фраза)`.
   Состав полей: машинный код, время регистрации (`get_rgtime`, `DateTime.Now.ToString()`),
   сведения (`get_rginfo`), срок/лимиты. Выписать точный порядок и какие поля под AES, какие под RSA.
7. **Ветки реестра** (HKLM, см. приложение) и порядок записи (`CreateSubKey`+`SetValue`),
   перекрёстные проверки `IsReg1`/`IsReg2`.
8. **Офлайн‑файл**: DES (`FrmRg.Decryptstr`, поля `desKey`/`desSIV`), формат содержимого.
9. **Парольные фразы AES** по каждому полю/каналу (`忘情水。。。`, `今天。。。`, `来生缘。。。`, `冰雨。。。`).

Артефакт: `SPEC.md` + тестовые векторы (вход → ожидаемый выход) для каждого преобразования,
снятые с **оригинального** клиента/сервера на стенде до перевыпуска ключей.

> Совет: значения `desKey`/`desSIV` и точный состав успешного ответа удобнее снять **динамически**
> (перехват в отладчике/инструментирование), чем только из IL.

## Фаза 2. Генерация собственной пары ключей (0.5 дня)

1. Сгенерировать RSA‑1024 (`e=65537`) — пару (n, d) на серверной стороне, надёжно хранить закрытый ключ.
2. Сериализовать **открытый** ключ в формат `ComponentKey` клиента:
   `Base64( [len(exp)] ‖ exp ‖ modulus )` (как у оригинала: первый байт = длина экспоненты, далее
   экспонента `01 00 01`, далее 128‑байтовый модуль). Проверить, что `RSAHelper.ResolveKey` корректно
   разбирает строку (длины совпадают, модуль 128 байт).
3. Подготовить серверную реализацию операций RSA, **совместимую** с клиентской `BigInteger`‑схемой
   (без паддинга, поблочно по 128, `ToHexString`, разделитель `@`, кодировка `Encoding.Default` при
   разбиении на блоки). Это критично: серверная подпись/шифрование должны байт‑в‑байт совпадать с тем,
   что ждёт клиент.

Артефакт: `keys/` (закрытый ключ — в защищённом хранилище), строка открытого ключа `ComponentKey`.

## Фаза 3. Перевыпуск ключей и пересборка клиента (0.5–1 день)

1. Взять распакованную пересобираемую сборку (пайплайн из задачи русификации).
2. **Заменить встроенный открытый ключ** `AwEAA…WyshiR` на наш во всех точках:
   `SR.GetMNum`, `SR.IsReg1/IsReg2`, `TCPClient.Connect`, `FrmRg.createinfo`, ресурсы `PKEY`/`G_SN`
   (ldstr‑операнды и `.resources`). Сделать единый патчер по сигнатуре строки.
3. **Перенаправить адрес сервера**: заменить зашифрованные `TCPClient.ip_address`/`ip_port`
   (переэшифровать новый IP/порт нашим открытым ключом — операция `m^e mod n`), либо пропатчить
   `Connect` на работу с открытым адресом/конфигом.
4. Сохранить обход self‑check (`code::HasShell → true`) и `PreserveAll` при пересборке (уже отлажено).
5. Опционально вынести адрес сервера в конфиг (`.settings`/файл), чтобы менять без пересборки.

Артефакт: `ZTool.exe` (re‑keyed), журнал замен (сколько ldstr/ресурсов/констант изменено).

## Фаза 4. Сервер активации: транспорт и протокол (2–4 дня)

1. **TCP‑слушатель** на выбранном порту (по умолчанию 58000).
2. **Парсер кадров**: читать 10 байт типа (LE int) + 10 байт длины (LE int) + тело; собрать тело
   по длине (учесть фрагментацию TCP).
3. **AES‑слой**: расшифровать тело `SecurityCenter`‑совместимо: `key=IV=MD5(Unicode(ToString(type)))`,
   CBC/PKCS7, вход — Base64. Получить полезную нагрузку запроса.
4. **Диспетчер по `Sendtype`**:
   - 128 `Apply_register` — заявка/проверка кода и привязки;
   - 130 `register` — выпуск лицензии;
   - 131 `verify_register` — проверка статуса;
   - 129 `Apply_Remove` / 132 `verify_Remove` — перенос/возврат лицензии.
5. **Формирование ответов**: те же строки‑статусы, что ждёт `SocketRecive`; при успехе — подписанный
   лицензионный блоб (Фаза 5). Ответ заворачивать в тот же кадр/AES, что и запрос.

Артефакт: серверный модуль `transport` + `protocol` с юнит‑тестами на кадрирование и AES (по векторам Фазы 1).

## Фаза 5. Серверная выдача лицензии в точном формате (2–4 дня)

1. **Расшифровать запрос**: развернуть `RSA` нашим **закрытым** ключом (клиент шифровал открытым) →
   получить код регистрации, машинный код, пароль.
2. **Бизнес‑правила**: проверить код в БД, лимит устройств, срок, привязку (machine code), статусы
   (занят/истёк/лимит) — вернуть соответствующий китайский статус при отказе.
3. **Сформировать лицензионный блоб** ровно в формате `IsReg1/IsReg2`:
   - собрать поля (машинный код, время, сведения, срок) в строку с разделителем `\n`;
   - наложить слои в правильном порядке: где требуется — **подписать** `RSA` закрытым ключом
     (клиент развернёт открытым), где требуется — `AES` (`SecurityCenter`, нужная фраза);
   - выдать значения для каждой из 4 веток реестра (учесть, что значения могут отличаться по ветке).
4. **Тест на реальном клиенте**: записать выданные значения в реестр тестовой машины и убедиться, что
   `IsReg1`/`IsReg2` принимают лицензию и снимают триал.

Артефакт: серверный модуль `licensing` (подпись + сборка блоба) + проверка на стенде.

## Фаза 6. Воспроизведение/верификация машинного кода (0.5–1 день)

1. На сервере реализовать разбор машинного кода: развернуть `RSA(закрытым)` → получить
   `UUID|disk|board[\r\n version]` (первые 117 символов) → использовать как идентификатор привязки.
2. Реализовать политику привязки: 1 код → N устройств, перенос освобождает слот.
3. Сверить алгоритм с эталоном: машинный код, выданный оригинальным клиентом на стенде, должен
   корректно разбираться сервером (по векторам Фазы 1).

Артефакт: модуль `machineid` + тесты.

## Фаза 7. Офлайн‑активация и перенос лицензии (1–2 дня)

1. **Офлайн‑файл**: сгенерировать содержимое, которое `FrmRg.Decryptstr` (DES, `desKey`/`desSIV`)
   корректно расшифрует и которое `IsReg` примет (привязка к машинному коду из QR/почты).
2. **Перенос (`转出授权`)**: реализовать серверную обработку `Apply_Remove`/`verify_Remove` — освобождение
   слота, опциональный «пароль защиты авторизации», очистка реестра на клиенте при успехе.
3. Покрыть пограничные случаи: «无需转出», «此电脑没有转出权限», «此注册码没有转出权限».

Артефакт: офлайн‑генератор + флоу переноса с тестами.

---

## Фаза 8. Интеграция реестра и кросс‑проверки (входит в Фазы 5–7)

1. Запись во **все** ветки HKLM (см. приложение) значениями нужного формата.
2. Гарантировать согласованность между `IsReg1` и `IsReg2` (они сверяют копии) — иначе «тихий» отказ.
3. Учесть права администратора (HKLM требует elevation) — отразить в инсталляторе/UX.

## Фаза 9. Канал обновлений (0.5 дня)

`CheckUpdate` (HTTP, `www.z-tool.cn`) может «перетереть» пропатченный клиент оригинальной сборкой.
Варианты: переориентировать на свой источник обновлений, либо отключить автообновление в клиенте.

## Фаза 10. Тестирование (2–3 дня)

- **Юнит**: RSA (наша пара) ↔ клиентская `BigInteger`‑схема (round‑trip по векторам); AES; кадрирование; DES.
- **E2E онлайн**: активация (валидный/невалидный код, занятый, истёкший, лимит, неверный пароль).
- **E2E офлайн**: машинный код → файл → активация.
- **Перенос**: активация на ПК‑A → перенос → активация на ПК‑B; повторные/без прав.
- **Реестр/anti‑tamper**: порча одной из веток → корректный отказ; перезапуск; смена «железа».
- **Регрессия SolidWorks**: загрузка надстройки `ZTool.dll` и базовые операции на пересобранном `ZTool.exe`.
- **Нагрузка/надёжность**: параллельные активации, обрывы TCP, таймауты.

Артефакт: тест‑план + автотесты + чек‑лист ручных сценариев.

## Фаза 11. Развёртывание и эксплуатация (1–2 дня)

1. Инфраструктура сервера (TCP 58000), мониторинг, логи, бэкап БД лицензий.
2. **Хранение закрытого ключа** (HSM/секрет‑менеджер) — компрометация = подделка любых лицензий.
3. Доступность: текущая схема — единая точка отказа; продумать HA/таймауты на клиенте.
4. Поэтапный rollout, миграция существующих лицензий (перевыпуск под новый ключ или двойная проверка).
5. Откат: оригинальный `ZTool.exe` — в истории git; план отката на старый сервер/ключ.

## Реестр рисков и митигации

| Риск | Влияние | Митигация |
|---|---|---|
| Несовпадение серверной RSA с клиентской `BigInteger`‑схемой (паддинг, кодировка, блоки) | Лицензия не принимается | Round‑trip тесты по эталонным векторам до интеграции |
| Неточный формат блоба/реестра | «Тихий» отказ `IsReg` | Снять эталон с оригинала на стенде, сверять побайтно |
| Перекрёстные проверки реестра рассинхронизированы | Отказ валидации | Писать все ветки атомарно, тест на порчу копий |
| Anti‑tamper/`HasShell`/обновление перетирает клиент | Откат к оригиналу | Сохранить обход, переориентировать/отключить `CheckUpdate` |
| Регрессия в SolidWorks из‑за пересборки `ZTool.exe` | Поломка надстройки | Регрессионное тестирование в реальном SolidWorks |
| Компрометация закрытого ключа | Подделка лицензий | HSM/секрет‑менеджер, ротация, аудит |
| Правовые риски | Юридические последствия | Подтвердить права/разрешение до старта |

## Критерии приёмки (Definition of Done)

- Пересобранный клиент с нашим ключом проходит онлайн‑активацию против нашего сервера и снимает триал.
- Офлайн‑активация и перенос лицензии работают по эталонным сценариям.
- `IsReg1`/`IsReg2` принимают лицензию; порча любой ветки реестра даёт корректный отказ.
- Надстройка `ZTool.dll` грузится и работает в SolidWorks без регрессий.
- Закрытый ключ хранится защищённо; есть мониторинг, бэкап и план отката.
- Автотесты на крипто/протокол/формат проходят на CI.

## Приложение A. Точные форматы (из аудита)

- **Сервер (текущий)**: `120.78.215.80:58000` (TCP).
- **Кадр**: `[int_to_Byte(type,10)] ‖ [int_to_Byte(len(body),10)] ‖ UTF8(AES_Base64.Trim())`;
  `int_to_Byte` — LE в 10 байт (значимы младшие 4). Ключ AES тела = `ToString((int)Sendtype)`.
- **Sendtype**: 128 `Apply_register`, 129 `Apply_Remove`, 130 `register`, 131 `verify_register`,
  132 `verify_Remove`.
- **Машинный код**: `RSA_pub( Mid("UUID|disk|board[\r\n ver]", 1, 117) )`, блоки 128, hex, разделитель `@`.
- **Запрос активации**: код = `L1-L2-L3-L4-L5`; в запрос входят `RSA(код)`, `RSA(пароль)`,
  машинный код, объединённые `\` и затем `RSA(...)`.
- **Лицензия в реестре**: поля через `\n`, слои `RSA`(подпись, разворот открытым ключом) + `AES`
  (`SecurityCenter`, MD5(фразы)); поля: машинный код, время (`DateTime.Now.ToString()`), сведения, срок.
- **Ветки реестра (HKLM)**: `Software\SolURxxCfNU\C4eHN4fjikBan`, `Software\SolURxxCfNU\HTwk2RCBDL`,
  `SOFTWARE\Microsoft\MzORu8qE4HhZ\Jlj4aG8uZBvW`, `SOFTWARE\Microsoft\MzORu8qE4HhZ\98CqyvBZcg`.
- **AES‑фразы**: `忘情水。。。` (машинный код), `今天。。。` (версия), `来生缘。。。` (перенос/сокет‑данные),
  `冰雨。。。` (маркер нативного модуля). Уточнить привязку фраза↔поле в Фазе 1.
- **DES (офлайн)**: `FrmRg.Decryptstr`, ключ/IV — поля `desKey`/`desSIV` (снять значения динамически).
- **RSA**: своя реализация `ZTool_rsa` (`e=65537`, 1024 бит, без паддинга, `ComponentKey` = Base64
  `[len(exp)]‖exp‖modulus`).

## Приложение B. Ключевые методы клиента
- Машинный код: `SR.GetMNum`, `GetUUID`, `GetDiskSN1_1/2_1`, `GetBoardId`, `MacAddress.GetNic`,
  `PhysicalDrive2.HardDisk.GetHddInfoNT`, `code.getver`, `code.GD51`.
- Валидация/запись: `SR.IsReg1`, `SR.IsReg2`, `get_rginfo`, `get_rgtime` (+ запись через `CreateSubKey`/`SetValue`).
- Крипто: `RSAHelper.EncryptString/DecryptString/ResolveKey/ComponentKey`, `BigInteger.modPow`,

# Расширенное техническое приложение

Ниже — углублённая проработка к фазам выше: референс‑алгоритмы сервера, протокол по сообщениям,
схема БД, шаблоны тест‑векторов, безопасность, миграция, наблюдаемость, CI и вехи. Все алгоритмы
приведены для построения **вашего** сервера под **вашу** пару ключей; форгинг лицензий под
оригинальный ключ не предполагается.

## Приложение C. Стек и принцип байт‑совместимости

Главный риск проекта — несовпадение серверной крипто‑реализации с клиентской по байтам. Минимизация:

- **Рекомендуемый стек: .NET (C#)**, чтобы **переиспользовать оригинальную `ZTool_rsa.dll`** (классы
  `RSAHelper`/`BigInteger`) и логику `SecurityCenter`. Тогда подпись/шифрование/паддинг/кодировки
  гарантированно совпадают с клиентом, и не нужно «переоткрывать» детали реализации RSA без паддинга.
- Если сервер на другом стеке (Go/Java/Python) — необходимо **точно воспроизвести**: `modPow` без
  паддинга, разбиение по 128 символов, `Encoding.Default` (системная ANSI‑кодовая страница) при
  GetBytes, `ToHexString()` формат, разделитель блоков `@`, MD5 от `Encoding.Unicode` (UTF‑16LE)
  для AES‑ключа. Любое расхождение кодовой страницы/паддинга ломает приём лицензии.
- Изоляция крипто в отдельный модуль `ztool-compat` с полным набором round‑trip тестов против
  эталонных векторов, снятых с оригинального клиента (Фаза 1).

## Приложение D. Серверные референс‑алгоритмы (псевдокод)

> Язык‑нейтральный псевдокод; в .NET — прямой вызов оригинальных классов.

### D.1 ComponentKey / ResolveKey (формат строки ключа)
```
ComponentKey(exp[], mod[]):
    buf = [ byte(len(exp)) ] ++ exp ++ mod
    return Base64(buf)

ResolveKey(s) -> (exp, mod):
    b = FromBase64(s)
    n = b[0]                 // длина экспоненты
    exp = b[1 .. 1+n]
    mod = b[1+n .. end]      // 128 байт для 1024 бит
```

### D.2 RSA без паддинга (совместимо с клиентским `BigInteger`)
```
EncryptString(text, key):            // m^e mod n, наружу — hex-блоки через '@'
    (exp, mod) = ResolveKey(key)
    out = ""
    for block in split(text, 128 символов):
        m = BigInteger( Encoding.Default.GetBytes(block) )
        c = modPow(m, exp, mod)
        out += c.ToHexString() + "@"
    return TrimEnd(out, '@')          // (клиент режет хвостовой разделитель)

DecryptString(text, key):            // c^e mod n (разворот подписи), та же экспонента
    (exp, mod) = ResolveKey(key)
    res = ""
    for hexblock in split(text, '@'):
        c = BigInteger.fromHex(hexblock)
        m = modPow(c, exp, mod)
        res += Encoding.Default.GetString(m.toBytes())
    return res
```
Ограничения клиента: `CheckSourceValidate`/`Mid(...,1,117)` — входной блок ≤ 117 символов
(иначе `"source string too long"`). Серверу учитывать тот же лимит при формировании полей.

### D.3 AES (эквивалент `SecurityCenter`)
```
key = iv = MD5( Encoding.Unicode.GetBytes(passphrase) )   // 16 байт
EncriptStr(plain, passphrase):
    return Base64( AES_CBC_PKCS7_Encrypt( Encoding.?.GetBytes(plain), key, iv ) )
DecriptStr(b64, passphrase):
    return GetString( AES_CBC_PKCS7_Decrypt( FromBase64(b64), key, iv ) )
```
BlockSize=128. Кодировку строки plaintext зафиксировать по эталону (Фаза 1).

### D.4 Кадрирование сокета
```
int_to_Byte(v, 10): 10-байтовый массив, младшие 4 байта = LE(v), остальные 0
sendFrame(type, dataPlain):
    body = UTF8( EncriptStr(dataPlain, ToString(type)).Trim() )
    frame = int_to_Byte(type,10) ++ int_to_Byte(len(body),10) ++ body
readFrame(stream):
    type = LE(read(10)[0..4]); len = LE(read(10)[0..4]); body = readExactly(stream, len)
    dataPlain = DecriptStr( UTF8.GetString(body), ToString(type) )
```

## Приложение E. Протокол по сообщениям (раскладка полей)

Заполнить точную раскладку на стенде (Фаза 1); ниже — каркас:

| Код | Имя | Запрос (после AES‑расшифровки) | Ответ (статус + данные) | Действие на клиенте |
|---|---|---|---|---|
| 128 | `Apply_register` | `RSA( код \ пароль \ машинный_код )` | статус: `无效注册码`/`注册码已过期`/`授权电脑数量已达上限`/`密码错误` или OK + параметры | проверка возможности активации |
| 130 | `register` | как 128 | `注册成功` + **лицензионный блоб** | запись блоба в реестр (см. F) |
| 131 | `verify_register` | `RSA( машинный_код )` | OK/`注册信息错误`/`注册码已过期` | проверка статуса при старте |
| 129 | `Apply_Remove` | `RSA( код \ машинный_код \ пароль )` | `转出授权成功/失败`, `无需转出`, `此电脑没有转出权限`, `此注册码没有转出权限` | очистка реестра при успехе |
| 132 | `verify_Remove` | как 129 | подтверждение переноса | освобождение слота |

Тексты статусов берутся `SocketRecive` буквально; сервер обязан возвращать те же строки.

## Приложение F. Лицензионный блоб и запись в реестр

- Запись (наблюдаемая в `SR`): поля объединяются разделителем `\t`, шифруются AES с фразой `今天。。。`
  (возможно с дополнительным RSA‑слоем) и пишутся в **несколько** мест:
  - **Информационные**: `HKLM\SOFTWARE\ZTool` → значения `sn` (машинный серийник), `AppDataPath`.
  - **Лицензионные (с кросс‑проверкой)**:
    `HKLM\Software\SolURxxCfNU\C4eHN4fjikBan` → `information`,
    `HKLM\Software\SolURxxCfNU\HTwk2RCBDL` → `information`,
    `HKLM\SOFTWARE\Microsoft\MzORu8qE4HhZ\Jlj4aG8uZBvW` → `F2S6qCdziIAm`,
    `HKLM\SOFTWARE\Microsoft\MzORu8qE4HhZ\98CqyvBZcg` → `information`.
- Чтение/валидация (`IsReg1`/`IsReg2`): значения снимаются, AES‑расшифровываются, RSA‑разворачиваются,
  делятся по `\n`/`\t`, сверяются машинный код и срок; копии сверяются между собой.
- Точный состав полей и порядок слоёв (что под `\t`, что под `\n`, где RSA, где AES) — **снять на стенде**:
  выдать оригинальным сервером лицензию, продампить значения реестра до/после, восстановить формат.
- Требуется запуск с правами администратора (ветки HKLM).

## Приложение I. Безопасность и хранение ключа

- Закрытый ключ — в HSM/секрет‑менеджере (KMS/Vault); сервер подписывает через API, ключ не покидает хранилище.
- Ротация ключей: таблица `server_keys` + поддержка нескольких активных `kid` на переходный период.
- Защита кодов: `protect_password_hash` (не хранить пароль в открытом виде).
- Логи без чувствительных данных; ограничение скорости активаций; защита от перебора кодов.

## Приложение J. Миграция существующих лицензий

- Поскольку клиент переключается на **наш** открытый ключ, ранее выданные (оригинальным ключом)
  лицензии перестанут проходить. Варианты:
  1. **Перевыпуск**: собрать базу действующих кодов/привязок, выдать новые блобы под наш ключ.
  2. **Двойная проверка на переходный период**: временно оставить в клиенте оба открытых ключа
     (оригинальный + наш) в `IsReg`, чтобы старые лицензии работали до перевыпуска. Требует доступной
     базы выданных лицензий или сбора машинных кодов.
- Коммуникация с пользователями: инструкция по повторной активации после обновления клиента.

## Приложение K. Наблюдаемость и эксплуатация

- Метрики: число активаций/отказов по статусам, переносы, латентность, ошибки парсинга кадров.
- Логи аудита (таблица `audit_log`), алерты на всплеск отказов/ошибок крипто.
- Runbook: недоступность сервера, исчерпание лимитов, инцидент с ключом, откат на старый сервер.
- Доступность: текущая схема — единая точка `IP:58000`; продумать HA, таймауты/ретраи на клиенте.

## Приложение L. CI/CD и автотесты

- Юнит: крипто round‑trip (D.2–D.4), парсер кадров, диспетчер сообщений, формат блоба.
- Контракт‑тесты: эталонные векторы (H) как фикстуры.
- Интеграционные: поднять сервер + эмулятор клиента (или реальный пересобранный) → онлайн/офлайн/перенос.
- Регрессия клиента: автоматическая проверка загрузки `ZTool.exe`/надстройки после пересборки.
- Артефакты сборки клиента (re‑keyed) подписываются и версионируются.
