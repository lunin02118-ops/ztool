# Методика тестирования генерации отчётов BOM

Документ описывает, как проверить, что экспорт спецификаций (BOM) из ZTool
работает корректно для всех типов отчётов. Экспорт физически выполняется
**внутри SolidWorks** (ZTool — аддон, который читает дерево сборки через API
SolidWorks), поэтому полноценная проверка возможна только на машине с
установленным SolidWorks и реальной русской моделью.

---

## 0. Предусловия

- Установлен SolidWorks, ZTool-аддон подключён, лицензия активна.
- Открыта **русская** сборка (`.SLDASM`), у компонентов заданы пользовательские
  свойства с именами из таблицы ниже (Имя детали, Номер чертежа, Материал …).
- `ZTool.settings` и папка `Шаблоны спецификации` доступны по абсолютному пути.

---

## 0.1 КРИТИЧНО: `<propname>` должен ТОЧНО совпадать с именами свойств модели

Экспорт сопоставляет данные **по точному имени пользовательского свойства**, а
не по позиции. Для каждой колонки ZTool ищет в модели свойство с именем, **в
точности** равным значению из `<propname>`. Совпадение строгое (любой лишний/
пропущенный символ → свойство не найдено → ячейка пустая).

> **Грабли, на которых мы потеряли время (зафиксировано).** Реальные свойства
> демо-модели называются **полными словами**: `设计`, `零件名称`, `图号`,
> `材料`, `类型`, `版本`, `表面处理`, `设计日期`, `重量`. В первых тестовых
> конфигах имена были по ошибке **обрезаны до первого иероглифа** (`设`, `零`,
> `图`, `材` …) → ни одно свойство не нашлось → все колонки пустые, хотя
> значения в модели заполнены. Вывод: **сверяйте `propname` посимвольно с тем,
> что реально показывает SolidWorks** (Файл → Свойства), а не «на глаз».

Поэтому:

- Перед прогоном откройте 1–2 детали → **Файл → Свойства** (вкладки
  «Настраиваемые»/«Свойства конфигурации») и выпишите **точные** имена свойств.
  `propname` в конфиге должен совпадать с ними символ-в-символ.
- Русский конфиг работает только на модели, где свойства названы **русскими**
  именами из таблицы §«соответствия» («Имя детали», «Номер чертежа» …). На
  китайской демо-сборке `0614-A00.SLDASM` нужен конфиг с её китайскими
  полными именами (см. `ZTool.demo-cn.settings`).
- Признак несовпадения имён: файл создаётся, шапка на месте, pre-flight (§1)
  = `PASS`, но **столбцы данных пусты**. Если бы дело было в шаблоне/анкерах —
  pre-flight показал бы `FAIL`.

> **Ограничение pre-flight.** Скрипт §1 проверяет только связку
> «конфиг ↔ шаблон» (анкеры). Он **НЕ** может проверить связку «`propname` ↔
> имена свойств в модели» — для этого нужен SolidWorks. Поэтому шаг сверки имён
> свойств выше — обязателен и выполняется вручную.

### Быстрая детерминированная проверка пайплайна на демо-модели

Если под рукой нет русской модели, проверить, что **сам механизм экспорта
работает**, можно на штатной китайской демо-сборке `0614-A00.SLDASM` с
конфигом под её китайские свойства:

```powershell
client-core\tools\set_bom_template_path.ps1 -Folder "D:\ваш\путь" -Settings "client-core\tools\ZTool.demo-cn.settings"
```

Подменить `ZTool.settings` на `ZTool.demo-cn.settings`, перезапустить ZTool,
открыть `0614-A00.SLDASM`, выполнить «Экспорт сводной спецификации». Ожидаемо:
столбцы данных **заполнены** значениями из модели. `ZTool.demo-cn.settings`
читает **полные** имена свойств демо-модели и пишет их в анкеры шаблона:

| Свойство модели (`propname`) | Анкер шаблона | Ячейка |
|---|---|---|
| `零件名称` (имя детали)   | `零名` | C6 |
| `图号` (номер чертежа)    | `图`   | D6 |
| `材料` (материал)         | `材质` | N6 |
| `类型` (тип)              | `类`   | H6 |
| `版本` (версия)           | `版`   | E6 |
| `表面处理` (обработка)    | `表处` | I6 |
| `重量` (масса)            | `重`   | J6 |

(`设计`/`设计日期`/`品牌` читаются, но в шаблоне нет анкера → не выводятся.)
Это доказывает, что пайплайн чтение→таблица→анкеры→запись строк работает; на
русском конфиге он так же заполнит данные на русской модели.

> Колонки `№ п/п`, `Эскиз`, `Имя файла`, `Путь` — служебные (генерируются
> ZTool, а не из свойств) и сопоставляются с анкером по **заголовку колонки
> в таблице ZTool**, а не по `propname`. Если они не заполняются — пришлите
> скрин строки заголовков таблицы ZTool: нужно сверить фактический текст
> заголовков с анкерами `序`/`缩图`/`磁文名`/`路`.

## 1. Pre-flight (офлайн, без SolidWorks)

Этот шаг ловит ошибки конфигурации (неверный путь к шаблону, отсутствующие
именованные диапазоны) **до** запуска SolidWorks:

```
python client-core\tools\check_bom_template.py "D:\ваш\путь\ZTool.settings"
```

Ожидаемо: `RESULT: PASS`. Проверяет:
- путь к шаблону у каждого пресета — абсолютный;
- каждый `<mappingname>` из `namemappinglist` существует как именованный
  диапазон в `bom_шаблон.xlsx`;
- наличие 8 режимов (пресетов) экспорта;
- для режимов с `ByRuler=true` — наличие ссылаемого правила в `FilterRulesList`;
- наличие 5 правил фильтрации в `FilterRulesList`.

Если `FAIL` — устранить до перехода к SolidWorks.

---

## 2. Перенацелить путь шаблона на свою папку установки

Если ZTool установлен не в `D:\ZTool`:

```powershell
client-core\tools\set_bom_template_path.ps1 -Folder "D:\ваш\путь" -Settings "D:\ваш\путь\ZTool.settings"
```

Скрипт перепишет `<bompath>` и `<bomname>` во всех пресетах. Перезапустить ZTool,
чтобы он перечитал настройки.

| --- | --- | --- | --- |

| Номер чертежа | PropVal_2 | `图` | **D6** |

| Тип | PropVal_4 | `类` | **H6** |

| Масса | PropVal_9 | `重` | **J6** |

## 4. Прогон 8 режимов экспорта

Для каждого пресета: меню ZTool → «Экспорт спецификации» → выбрать пресет →
сохранить → открыть полученный `.xlsx` и сверить с критериями.

Порядок пресетов в таблице **строго совпадает** с порядком `<bomsetting>` в
`ZTool.settings` (и с `MODES` в `validate_bom_exports.py`). Эталонные числа строк
приведены для демо-сборки `0614-A00.SLDASM` (29 компонентов в сводной).

| # | Пресет (русский) | type | Эскизы | Фильтр | Что ожидать (демо 0614-A00) |
| --- | --- | --- | --- | --- | --- |
| 1 | Экспорт сводной спецификации | 0 | нет | — | Плоский список всех компонентов; одинаковые свёрнуты, в Кол-во (G6…) — суммарное количество. **≈29 строк.** |
| 2 | Экспорт иерархической спецификации | 1 | нет | — | Состав по уровням сборки (узлы и их детали), вложенность. **≥ сводной (≈32).** |
| 3 | Экспорт спецификации верхнего уровня | 2 | нет | — | Только прямые дочерние (Level 1), без раскрытия подсборок. **< сводной (≈6).** |
| 4 | Экспорт сводной спецификации деталей | 3 | нет | — | Только детали (.SLDPRT), без подсборок. **≤ сводной (≈25).** |
| 5 | Экспорт сводной спецификации (с эскизами) | 0 | да | — | Как №1 + в Эскиз (M6…) вставлены миниатюры. **≈29 строк + картинки.** |
| 6 | Экспорт иерархической спецификации (с эскизами) | 1 | да | — | Как №2 + эскизы в M-колонке. **≈32 строки + картинки.** |
| 7 | Обрабатываемые детали | 0 | нет | Тип ∋ {机加, Мех.обработка, Листовая, Литьё, Сварка} | Только обрабатываемые детали. На демо `Тип`=机加 → **≈15 строк** (строго меньше сводной). |
| 8 | Покупные изделия | 0 | нет | Тип ∋ {外购, Покупное, Стандартное} | Только покупные/стандартные. На демо `Тип`=外购 → **≈9 строк** (строго меньше сводной). |

> **Важно про фильтры (режимы 7–8):** фильтрация по правилу применяется **только
> при `type=0`** (так устроен экспорт в ZTool.exe). Поэтому фильтрованные пресеты
> сделаны сводными (`type=0`, `ByRuler=true`, `ByFilter=false`). Сумма строк
> 7 (обрабатываемые) + 8 (покупные) + сборки (组件) ≈ сводная (15+9+5=29).

### 4.1 Редактирование правил фильтрации

Значения в правилах (Мех.обработка, Покупное и т.д.) — **заглушки**. Замените их
на реальные значения свойства «Тип» из ваших моделей.

Два способа:
1. **Через ZTool UI:** Спецификация → «Настроить схему спецификации» →
   выбрать правило → редактировать значение фильтра.
2. **Через XML:** открыть `ZTool.settings`, найти `<FilterRulesList>` → в блоке
   нужного `<CustomRule>` изменить значение после последнего `|@#$%|` в строке
   `<RuleList>`. Значения разделяются `;`.

## 5. Критерии приёмки (PASS/FAIL)

PASS, если для каждого пресета:
- файл создаётся без ошибки «не удалось найти файл …\Documents\…» (путь
  абсолютный отработал);
- заполнена **не только шапка**, а строки данных (с строки 7 вниз) по числу
  компонентов модели;
- **№ п/п** (A) — последовательная нумерация 1…N;
- **Кол-во** (G) — числа > 0;
- **Свойства** (C, D, E, H, I, J, N) — заполнены значениями из модели
  (если propname совпадает);
- **Путь** (O) — абсолютные пути к файлам деталей;
- для пресетов «с эскизами» — в Эскиз (M) присутствуют картинки;
- для фильтрованных режимов (7, 8) — список содержит **строго меньше** строк,
  чем сводная (режим 1), и только подходящие по правилу детали. Если строк
  столько же, сколько в сводной, — **фильтр НЕ применился** (это FAIL, см.
  «формат правил» ниже). Пустой результат означает, что значения правила не
  совпали ни с одним значением свойства «Тип» в модели — поправьте значения;
- иерархия/верхний уровень/только-детали — соответствуют типу.

FAIL фиксировать с деталями: имя модели, какой пресет, скрин Excel, какие
ячейки пусты, текст ошибки (если была).

---

## 6. Что приложить к отчёту

- Имя тестовой модели и число компонентов.
- По каждому из 8 пресетов: полученный файл (или скрин листа `图明1`).
- Любые пустые/неверные колонки — со ссылкой на строку таблицы §3.

## 7. Локализация: русский / английский

Архитектура поддерживает будущую английскую версию. Эталонные файлы:
- `locale/ru/bom_modes.yaml` — русские режимы, описания, правила
- `locale/en/bom_modes.yaml` — английские эквиваленты (заготовка)

Для переключения языка: замените имена в `ZTool.settings` `<bomsettings>` и
`<FilterRulesList>` на соответствующие из `locale/en/bom_modes.yaml`,
обновите `<propname>` под английские имена свойств моделей.

### Формат строки правила (критично!)

Каждая строка `<RuleList>` имеет **ровно три поля**, разделённые
**ТАБУЛЯЦИЕЙ + `|@#$%|`** (символ табуляции `\t`, затем `|@#$%|`):

```
$ИмяСвойства$<TAB>|@#$%|Оператор<TAB>|@#$%|значение1;значение2
```

- **Разделитель — табуляция, не пробелы.** Если поставить пробелы, ZTool.exe не
  распознает поля, правило молча игнорируется, и экспортируется **полный список**
  (это и был баг режимов 7–8).
- **Оператор — локализованная русская строка**, ровно одна из:
  `Равно`, `Не равно`, `Содержит`, `Не содержит`, `Начинается с`,
  `Не начинается с`, `Заканчивается на`, `Не заканчивается на`.
  Китайские коды (`包含`, `等于`, …) в русифицированной сборке **не работают** —
  их быть не должно.
- **Имя свойства:** `$Имя$` — сопоставляется со столбцом разрешённого значения
  (PropResolvedVal, как в Excel-экспорте); `%Имя%` — со столбцом сырого значения;
  `<Имя>` — со служебным столбцом.
- Несколько значений в поле значения разделяются `;` (логика «любое из»).

При переходе на EN меняются: имя правила, имя свойства (`$...$`) и значения.
**Операторы переводятся на английские локализованные строки** (а не остаются
китайскими). Проверяйте формат правил пред-флайтом
`check_bom_template.py` — секция `[1c]` ловит неверный разделитель/оператор.
