## 0. Принципы исполнения

1. **Паритет — закон.** Любой реконструированный модуль должен вести себя как оригинал
   (A/B на тест-сборке `TestModel/0614-A00.SLDASM`), пока не доказано обратное и не согласовано.
2. **Гейты вместо ручных проверок.** Каждая ручная проверка, ловящая регресс, становится скриптом/тестом.
3. **Роль строки важнее языка.** internal keys / commands / Excel-anchors / enum-tags — НЕ переводятся.
4. **Маленькие PR, по одному модулю/подсистеме**, каждый — с implementation report в `docs/audit/`
   и прохождением `docs/production/AUDIT_GATES_RU.md`.
5. **Реконструкция идёт за фасадом.** Пока модуль не доказан, остаётся бинарный fallback
   (reinject), чтобы `main` всегда был отгружаемым.
6. **Никакого force-push в `main` без явного разрешения пользователя** (актуально для очистки LFS-истории).

## A. Гигиена и доступность репозитория (P0) 🟢

A1. **Вынести memory-dump из Git.** 🟢 — **СДЕЛАНО без force-push (PR #34).**
- Было: `dumps/full-memory-20260609-081854/SLDWORKS_1344_full.dmp.part00{1,2}` (1.6+1.5 ГБ, LFS),
  чистый `git clone` падал: `This repository exceeded its LFS budget` (воспроизведено аудитором).
- Решение (PR #34): дамп убран из **текущего** коммита обычным `git rm` (история `main` НЕ переписана,
  force-push не нужен). Обычный `git clone` тянет LFS-объекты только для HEAD — после удаления из HEAD
  smudge больше не запускается. Проверено свежим clone: clone проходит, `git lfs ls-files` пуст.
  `dumps/.../README.txt` сохранён как карточка восстановления (SHA256 + размеры + способ воспроизведения
  дампа на машине с SW). Дамп архивируется во внешнем хранилище (адрес вписать в README после загрузки).
- DoD: ✅ свежий `git clone` без smudge-ошибок; ✅ CI зелёный.
- Опционально (отложено): полная вычистка 3 ГБ из `.git` через `git filter-repo`/BFG — **требует force-push
  в `main`**, делать только при явном разрешении и только ради размера репозитория; на доступность clone
  уже не влияет.

## C0. Решение проблемы тестирования у исполнителя (P0) — реализовано PR #35

Симптом (с машины SW 2025): SolidWorks/ZTool из автоматизации-shell давали ложную ошибку
«Не удалось загрузить Microsoft .NET Framework» / вечный splash, либо грузили **не тот** `ZTool.dll`,
либо репозиторий не клонировался из-за LFS. Три корневые причины и их закрытие:

1. **LFS ломает clone/pull.** → A1 / PR #34 (или обходной `GIT_LFS_SKIP_SMUDGE=1 git clone`).
2. **Пустой `WINDIR`/`SystemRoot` в окружении запуска** → ложная ошибка .NET / splash, хотя через ярлык
   всё работает. → `scripts/sw_test_preflight.ps1` нормализует `WINDIR/SystemRoot/ComSpec` и проверяет RegAsm.
3. **Грязный реестр (stale `RegAsm CodeBase`/AddIn-ключи)** → SW грузит чужой DLL. → тот же скрипт бэкапит
   реестр, сообщает о stale-ссылках и с `-Register` перерегистрирует текущий `runtime\ZTool.dll` и проверяет
   `CodeBase`; плюс сверяет SHA256 `ZTool.exe`/`ZTool.dll` с принятыми.

Инструменты: `scripts/sw_test_preflight.ps1` (идемпотентный, read-mostly, JSON-отчёт) и skill
`.agents/skills/solidworks-ztool-testing` (чеклист для будущих сессий-исполнителей). Это автоматизирует
ручной runbook из `docs/release/FULL_TEST_METHODOLOGY_RU.md` §2.1/§2.2.

DoD исполнителя (🟠): `sw_test_preflight.ps1 -RuntimeDir .\runtime -Register` → status `PASS`; затем
`TestModel/0614-A00.SLDASM` открыт через Explorer, ZTool из ленты, окно развёрнуто, хеш работающего
`ZTool.exe` совпадает — и только из такой сессии запускать BOM/цвет/свойства.

## D. Целевая архитектура решения (target solution) 🟢

Единый `ZTool.sln` со слоями (имена — предложение, привести к конвенции):

```
src/
  ZTool.Core/           # домен: модель документа/свойств, BOM-движок, конфиг (без UI, без SW)
  ZTool.SolidWorks/     # тонкий слой над SolidWorks API (interop), фасады чтения сборок/свойств

  ZTool.Localization/   # ресурсы RU/EN (.resx), каталог строк по ролям, провайдер строк

tools/                  # build/patch-инструменты (постепенно отмирают по мере реконструкции)
tests/
  ZTool.Core.Tests/
  ZTool.Licensing.Tests/
  ZTool.Bom.Tests/

license-server/         # существующий Python-сервер (отдельный деплой)
```

(решить отдельно — зависит от SolidWorks interop и WinForms). DoD: `ZTool.sln` собирается из

## F. Стратегия декомпиляции → чистый исходник 🟢 (валидация 🟠)

F1. **Снять рабочие сборки.** Использовать уже распакованный (de-ConfuserEx) `ZTool.exe` из ветки
русификации и дамп `ZTool.dll` (`dumps/manual-test-20260609-081854/ZTool_dumped.dll`,
`dumps/candidate-ru-20260609/*`). DoD: зафиксированы SHA256 входов реконструкции.

F2. **Декомпиляция.** ILSpy / dotPeek / dnSpyEx → C#-проект на модуль. Стартовать с самого
независимого слоя (Core/Licensing), затем формы, затем add-in. DoD: каждый модуль компилируется.

F3. **Очистка декомпилята** (это и есть «лёгкая поддержка»):
- убрать артефакты VB (`Microsoft.VisualBasic`, `ProjectData`, `Operators`, `Conversions`) → идиоматичный C#;
- переименовать CN-идентификаторы (enum/методы) в EN с сохранением сериализуемых значений как
  атрибутов/маппинга (НЕ менять то, что уходит в `ZTool.settings`/протокол как строка — см. §H);
- разнести «божественные» формы (`Frmmain`) на view + логику (MVP/MVVM-lite), вынести домен в `ZTool.Core`;
- заменить захардкоженные шифр-константы адреса/порта на конфиг (`ZTool.Licensing` читает endpoint
  из настроек/инсталлятора, см. `client-rekey` — больше не нужен IL-патч).
DoD: на модуль — ревью, что нет VB-артефактов и CN-идентификаторов вне whitelisted сериализации.

F4. **Ресурсы и формы.** WinForms designer-файлы восстановить из встроенных `.resources`; строки
вынести в `.resx` (см. §H). Ribbon: перенести с Win32 Windows Ribbon Framework на нативную
SolidWorks Command Manager/Command tab (add-in рисует ленту через SW API) — это убирает хрупкие
`RT_STRING`-патчи. DoD: лента и формы рисуются из исходника, иконки на месте (🟠 проверка в SW).

F5. **COM add-in под SolidWorks API.** `ZTool.AddIn` реализует `ISwAddin`, регистрируется через
`RegAsm`/`COMRegister`; интеграция через `SolidWorks.Interop.*` (правильные версии под SW 2025).
Убрать хак выбора exe по `AssemblyName` и рукопожатие `Getpkt()` — раз мы владеем обеими сторонами,
IPC заменяется на нормальный контракт (in-process вызовы или явный named-pipe с версионированием).
DoD: add-in грузится в SW 2025 без reinject; чтение сборки даёт N позиций (🟠 parity vs эталон).

F6. **Порядок реконструкции и чекпойнты паритета.** На каждый модуль: реконструировать → собрать →
заменить соответствующую часть → прогнать `client-semantic-parity` (C4) → зафиксировать PASS.
Рекомендуемый порядок: `ZTool.Core`(config/props) → `ZTool.Licensing` → BOM-движок →
`ZTool.SolidWorks` чтение → формы → add-in/лента → апдейтер. DoD: после каждого шага parity PASS.
