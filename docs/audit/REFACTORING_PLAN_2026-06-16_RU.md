# План рефакторинга и подготовки к продакшену ZTool

- Дата: 2026-06-16
- Вход: `docs/audit/CODEBASE_AUDIT_2026-06-16_RU.md`
- Аудитор: Devin. Исполнитель: отдельный агент на машине с **SolidWorks 2025**.
- Правовое основание: `docs/legal/ztool_license_agreement_signed.pdf`.

> Назначение документа — дать исполнителю проверяемый, пошаговый план: что делать, в каком
> порядке, чем доказывать «готово». Каждая задача имеет **Definition of Done (DoD)** и метку,
> можно ли её закрыть без SolidWorks. Метки: 🟢 = без SW (любая машина), 🟠 = требуется SW 2025.

---

## 0. Принципы

1. **Не ломать поведение vendor-бинаря.** Любая правка `ZTool.exe`/`ZTool.dll` — через
   существующий pipeline (`client-core`), с `Reinjector --verify` и нулевыми dangling refs.
2. **Гейты вместо ручных проверок.** Каждый ручной чек, который реально ловит регресс,
   превращаем в исполняемый скрипт/тест.
3. **Роль строки важнее языка.** Никогда не переводить internal keys/commands/anchors.
4. **Сначала доступность и доказуемость, потом расширение функций.** Порядок фаз: A → B → C → D.
5. **Маленькие PR по одной фазе**, каждый с implementation report в `docs/audit/` и прохождением
   `docs/production/AUDIT_GATES_RU.md`.

## Фаза C. Доказуемость клиента — Phase 11 (P0/P1)

Это центральная фаза перед продакшеном. Большая часть — 🟠 (нужен SW 2025).

C1. **`internal-string-invariants` тест.** 🟢
- Сравнивать оригинал и локализованный `ZTool.exe`/`ZTool.dll` по whitelisted internal keys.
- Для `零件` проверять не просто наличие, а конкретные методы `Frmmain`
  (`GetMaterialName`, `Material_list_ItemClicked`, `mens_click`, `addctl`, `readstr`).
- Любой перевод vendor-key без whitelist → fail.
- DoD: тест в CI (`client-core-windows.yml`), падает на намеренно «сломанном» переводе.

C2. **`solidworks-registry-preflight.ps1`.** 🟠
- Проверять `RegAsm CodeBase`, global/versioned `HKLM Addins`, global/versioned
  `HKCU AddInsStartup`; печатать SHA256 реально используемых `ZTool.dll`/`ZTool.exe`;
  падать при чужом `ZTool.exe`/старом пути/китайском описании.
- DoD: скрипт в репо; на тест-машине PASS зафиксирован в `manual-test-reports/`.

C3. **Единый способ запуска приёмки.** 🟠
- Открывать `TestModel/0614-A00.SLDASM` через Explorer/shell association; ZTool запускать только
  кнопкой add-in из ленты. Прямой `SldWorks.exe` для приёмки запрещён, пока не доказана
  эквивалентность. DoD: зафиксировано в методике и в preflight.

C4. **`client-semantic-parity` прогон (A/B против оригинала).** 🟠
- Workflow: чтение N позиций (целевое — 29 на тестовой сборке), BOM 8/8 режимов, назначение
  материала, ручной цвет, случайный цвет, save→SW→reread, сравнение цвета ячейки и детали.
- Результат: машинно-читаемый JSON + скриншоты + Excel-файлы в `manual-test-reports/`/`full-auto-runs/`.
- DoD: все пункты PASS; нет CJK в видимых путях/названиях (кроме разрешённых anchors/keys);
  Event Viewer/WER без новых crash.

C5. **Полуавтоматизация методики.** 🟠
- Макрос/PowerShell: открыть модель, записать русские свойства (`WriteRussianProps.vbs`),
  вызвать ZTool через add-in, выполнить экспорт режимов, собрать артефакты, хеши, registry-state.
- DoD: один запуск даёт полный пакет доказательств; ручные шаги — только там, где клик ненадёжен.
