# Аудит разрыва production-hardening и фактической готовности клиента

Дата: 2026-06-16

База аудита: `main` @ `de51c38` (`fix(client): preserve part kind key for material colors`)

## Короткий вывод

Претензия справедливая: большая работа по production-hardening была реальной, но она не равна полноценному восстановлению поддерживаемого исходного кода ZTool-клиента. Фазы #13-#22 в основном закрывали сервер лицензий, CI, упаковку, безопасность, manifest/hash gates и воспроизводимость бинарной сборки. Они не давали автоматического доказательства паритета поведения `ZTool.exe`/`ZTool.dll` с оригинальной китайской версией внутри SolidWorks.

Текущая клиентская часть остается гибридом:

- частично поддерживаемый C# в `client-core/src`;
- бинарная пересборка через `Publicizer`/`Reinjector`;
- локализация и точечные IL/resource-патчи через `Localizer`, `BinderInject`, `PmpGuardPatch`, `NullModalGuard`;
- обязательные живые проверки в SolidWorks для семантики UI/COM/материалов/цветов.

Это можно довести до production, но нельзя честно считать "обычным поддерживаемым исходным кодом" до отдельной фазы реконструкции/владения исходниками клиентского UI и add-in DLL.

## Что было реально закрыто

Закрыто хорошо:

- сервер лицензий: секреты, логирование, протокол, DB state, activation/transfer state, deploy/backup/monitoring;
- CI и security gates: secret scan, Windows client build smoke, localization scan, release manifest smoke;
- packaging: `manifest.json`, `SHA256SUMS.txt`, запрет лишних файлов, проверка `SolidWorksTools.dll`, `MyMaterials.sldmat`, `materialpath`, `usematerialcolor=true`;
- build discipline: input hash gate, `Reinjector --verify`, `dangling typerefs = 0`, fail-closed на missing target type/method;
- документация: full functional test methodology с A/B-паритетом против оригинала.

Прямое свидетельство из репозитория:

- `docs/audit/phase-08-client-core-reinjector-hardening-implementation-report.md` фиксирует, что Phase 08 проверяет Reinjector/build, но SolidWorks smoke не выполнялся.
- `docs/audit/phase-10-release-packaging-implementation-report.md` прямо говорит, что packaging не заменяет manual SolidWorks acceptance.
- `docs/release/FULL_TEST_METHODOLOGY_RU.md` уже требует A/B-паритет, BOM 8/8, цвета материалов/покраски и запуск через файл сборки, но это методика ручной/полуавтоматической приемки, а не CI-гейт.

## Что не было закрыто

Не было автоматического доказательства следующих вещей:

1. Что все внутренние китайские строки являются видимыми UI-строками.
2. Что локализация не трогает строки-ключи, используемые кодом как enum/tag/filter.
3. Что `ZTool.exe` запускается именно тем способом, которым его запускает add-in из SolidWorks.
4. Что реестр SolidWorks содержит полный набор discovery/startup ключей, а не смесь старых и новых записей.
5. Что ручной/случайный цвет, цвет материала и внешний вид детали меняются на том же уровне SolidWorks API, что в оригинале.
6. Что `ZTool.settings` после чистки не потерял функциональные defaults, которые не выглядят как "данные пользователя", но являются частью поведения.

## Почему прошел баг с цветами

Причина: строка `零件` была ошибочно воспринята как переводимая UI-строка "Деталь", хотя в части методов `Frmmain` это внутренний ключ `Col_Extname.Tag`.

Фактическая цепочка:

- `translations.tsv` содержит перевод `零件 -> Деталь`.
- После локализации часть логики цветов/материалов сравнивала tag уже с `"Деталь"`, а остальные данные/контекст ожидали vendor-key `"零件"`.
- Команды вроде "Случайный цвет" и ручная покраска работали неполно или пропускали выбранные строки.
- Первый фикс восстановил ключ только в части методов.
- Второй фикс `de51c38` расширил восстановление на все найденные методы `Frmmain`: `GetMaterialName`, `Material_list_ItemClicked`, `mens_click`, `addctl`, `readstr`.

Почему gate не поймал:

- `localization_scan.py` проверял наличие неклассифицированных Han-строк, но не различал "видимый текст" и "семантический ключ".
- Не было теста "внутренние ключи после локализации совпадают с оригиналом".
- Не было живого automated/parity smoke: назначить материал, задать случайный цвет, сохранить в SW, перечитать, сравнить цвет ячейки и детали с оригиналом.

Итог: это не только баг локализации. Это дефект процесса: строки в бинаре классифицировались по языку, а не по роли.

## Почему прошла проблема запуска SolidWorks/.NET

Причина: не был зафиксирован один допустимый способ запуска для тестов и не было registry preflight.

Фактическая цепочка:

- При прямом запуске SolidWorks/через shell helper появлялась ошибка "Не удалось загрузить Microsoft .NET Framework".
- При открытии тестовой сборки `.SLDASM` из проводника поведение отличалось и работало корректнее.
- В реестре важны не только COM `CodeBase` от `RegAsm`, но и discovery/startup ключи SolidWorks:
  - `HKLM\SOFTWARE\SolidWorks\Addins\{59959DFA-3229-4B86-852E-52ABF2BDB8C0}`
  - `HKLM\SOFTWARE\SolidWorks\SOLIDWORKS 2025\Addins\{...}`
  - `HKCU\SOFTWARE\SolidWorks\AddInsStartup\{...}`
  - `HKCU\SOFTWARE\SolidWorks\SOLIDWORKS 2025\AddInsStartup\{...}`
- Чистка реестра без восстановления полного набора ключей может ломать обнаружение add-in, а сохранение старых ключей может приводить к запуску не той сборки/старому описанию.

Почему gate не поймал:

- release verifier проверяет файлы пакета, но не реестр установленной машины.
- manual methodology уже говорит "открывать через файл сборки", но это появилось как знание из отладки, а не как enforceable script.
- Не было `preflight`-скрипта, который перед тестом печатает: какой `ZTool.dll` зарегистрирован, откуда запускается `ZTool.exe`, какие add-in ключи активны, какой процесс реально поднят.

## Почему hardening не дал "полноценный исходный код"

Текущая архитектура клиентской сборки прямо описана в `client-core/README.md`: оригинальный `ZTool.exe` publicize -> компиляция частичного `ZTool.Core.dll` -> reinject method bodies обратно в копию оригинального exe.

Это поддерживаемый build pipeline, но не полноценное владение исходниками приложения.

Ограничения такого подхода:

- методы, которых нет в `client-core/src`, остаются vendor-бинарем;
- формы, меню, часть бизнес-логики и add-in DLL правятся через ресурсы/IL-патчи;
- поведение может зависеть от строк, metadata tokens, obfuscation patterns, registry side effects;
- обычные unit-тесты покрывают только реконструированные части, а не весь фактический runtime.

Отсюда ощущение "снова патчи": оно технически верное. Мы сделали патчи более воспроизводимыми и проверяемыми, но еще не превратили весь клиент в нормальный исходный проект.

## Почему это оказалось в production-train

Основная ошибка управления качеством: acceptance был раздроблен.

- Server/security phases имели хорошие автоматические критерии.
- Client binary phases имели build/hash/typeref критерии.
- Packaging имел manifest/hash/forbidden-file критерии.
- Но SolidWorks functional parity остался как отдельная методика, не как обязательный blocked merge gate.

В результате PR мог быть "APPROVED FOR MERGE" по своему scope, но общая production readiness не была доказана для клиента.

Это не значит, что работа была бесполезной. Это значит, что слово "production-ready" применялось к инфраструктурным слоям раньше, чем был закрыт живой клиентский паритет.

## Риск-регистр по итогам аудита

| Риск | Суть | Статус |

| R-C1 | Внутренние vendor-строки переводятся как UI и ломают логику | Частично закрыто для `零件`, нужен общий gate |
| R-C2 | Запуск SW через неправильный shell/context дает ложные .NET ошибки | Не закрыто системно, нужен launch/registry preflight |

| R-C4 | Материалы/цвета имеют несколько API-уровней: material library, component appearance, document appearance | Частично проверено вручную, нужен parity test |
| R-C5 | Клиент не является полностью исходным кодом; часть поведения остается бинарем | Принято как архитектурный риск, но должно быть явно зафиксировано |
| R-C6 | Ручные отчеты не превращены в исполняемые регрессии | Не закрыто |

## Что надо сделать перед production GO

Минимум, без переписывания всего клиента:

1. Ввести `client-semantic-parity` gate.
   - A/B против оригинала по ключевым workflow: чтение 29 позиций, BOM 8/8, материалы, ручной цвет, случайный цвет, сохранение в SW, перечитывание.
   - Результат: машинно-читаемый JSON + скриншоты + Excel-файлы.

2. Добавить `internal-string-invariants` test.
   - Сравнивать оригинал и локализованный exe по whitelisted internal keys.
   - Для `零件` проверять не просто наличие, а конкретные методы `Frmmain`.
   - Любой новый перевод vendor-key без whitelist должен падать.

3. Добавить `solidworks-registry-preflight.ps1`.
   - Проверять `RegAsm CodeBase`.
   - Проверять global/versioned `HKLM Addins`.
   - Проверять global/versioned `HKCU AddInsStartup`.
   - Печатать SHA256 `ZTool.dll` и `ZTool.exe`, которые реально используются.
   - Падать, если найден другой `ZTool.exe`/старый путь/китайское описание.

4. Зафиксировать единственный способ запуска теста.
   - Открывать `TestModel/0614-A00.SLDASM` через shell association/Explorer.
   - ZTool запускать только кнопкой add-in из ленты SolidWorks.
   - Прямой запуск `SldWorks.exe` считать недопустимым для приемки, пока не доказана эквивалентность.

5. Перевести ручную методику в полуавтомат.
   - Макрос/PowerShell должен открыть модель, записать русские свойства, вызвать ZTool через add-in, выполнить экспорт режимов, собрать артефакты.
   - Где нельзя кликнуть надежно, оператор делает шаг, но скрипт фиксирует процессы, хеши, registry state, выходные файлы.

6. Разделить статусы:
   - `hardening complete` — инфраструктурно/пакетно готово;
   - `client parity complete` — подтверждено в SolidWorks;
   - `production GO` — только после обоих.

## Стратегический выбор

Есть два честных пути.

### Вариант A: оставить бинарный pipeline, но сделать его строгим

Плюсы: быстрее, не надо полностью переписывать приложение. Минусы: всегда будет требовать A/B-паритет, IL-diff, string-invariant gates и живые SolidWorks-тесты.

Это реалистичный путь для ближайшего релиза.

### Вариант B: реконструировать полноценный исходный клиент

Плюсы: нормальная поддерживаемость, обычные тесты, меньше скрытых vendor-key ловушек. Минусы: дорого и долго; нужно декомпилировать/перенести формы, ресурсы, add-in DLL, COM-логику, BOM/export, материалы/цвета, licensing UI и прогнать полную совместимость.

Это правильный долгосрочный путь, если проект будет жить и развиваться.

## Рекомендация

Не объявлять текущую ветку production-ready только по hardening-фазам. Считать ее "release-candidate infrastructure ready".

Следующий обязательный этап перед выпуском: отдельная фаза `Phase 11 — client semantic parity and SolidWorks automation`.

Критерий закрытия Phase 11:

- чистый runtime из release package;
- registry preflight PASS;
- запуск только через тестовую сборку `.SLDASM`;
- чтение 29 позиций PASS;
- BOM 8/8 PASS;
- русские свойства PASS;
- материал/ручной цвет/случайный цвет/сохранить в SW/перечитать PASS;
- нет CJK в видимых путях/названиях, кроме явно разрешенных Excel anchors/internal keys;
- Event Viewer/WER без новых crash;
- отчет и артефакты лежат в `manual-test-reports/` или `full-auto-runs/`.

До этого момента корректная формулировка статуса: "hardening train merged, client production acceptance not complete".
