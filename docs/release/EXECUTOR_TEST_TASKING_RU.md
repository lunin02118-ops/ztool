# Наряд-задание агенту-исполнителю: полное тестирование SWTools (from-source) + отчёт

**Кому:** агент-исполнитель на **реальной машине с SolidWorks 2025** (валидное железо + донгл,
сеть до боевого сервера лицензий).
**Статус повода:** отчётов и доказательств по предыдущему прогону нет; ранее отмечены только
S1/S3 частично, а L1–L5 **не выполнены** (стенд был грязный — оставались `HKCU\SOFTWARE\SWTools\sn`
и старый `HKCU\SOFTWARE\ZTool\sn`). Нужно выполнить тест **с нуля** и сдать отчёт.

## 0. Контекст (что тестируем и почему)
- Объект: `ZTool.exe` под брендом **SWTools**, собранный **из исходников** `client-src`
  (`dotnet build client-src/ZTool.csproj -c Release` → `client-src/bin/Release/net48/ZTool.exe`),
  рядом нативная `SWToolsARM.dll`. Для сверки 1:1 — **боевой** `SWTools.exe` (бинарный пайплайн).
- Зачем: подтвердить, что from-source клиент ведёт себя 1:1 с боевым (UI/ребренд/локализация,
  лицензирование, экспорт BOM) на реальном железе. На облачной VM это невозможно: аппаратный
  гейт (`SR.Isme`/`GetMNum`) рубит запуск ещё до сети — так же ведёт себя и боевой exe.
- Главная цель проекта: понятные исходники, из которых быстро собирать/проверять релизы. Этот
  тест — приёмка текущего состояния from-source.

## 1. ОБЯЗАТЕЛЬНО перед запуском SolidWorks
Используй skill **`solidworks-ztool-testing`** (репо `ztool`) — он снимает повторяющиеся сбои
запуска SW («failed to load Microsoft .NET Framework», зависание на сплэше, загрузка не той
`ZTool.dll`). Без него живые SW-сценарии не начинать.

## 2. Гейты сборки (CI-эквивалент локально) — ДО живых тестов
Выполнить и приложить вывод; все должны быть **PASS**:
```
dotnet build client-src/ZTool.csproj -c Release
python tools/check_source_string_invariants.py --self-test
python tools/check_source_string_invariants.py --root client-src
python tools/check_no_cjk_filenames.py
python client-core/tools/check_bom_template.py SWTools.settings
python client-core/tools/check_bom_template.py client-core/dist/SWTools.settings
python tools/bom_export_assert.py --self-test
python -m pytest license-server -q
python tools/secret_scan.py
```
Записать `ProductName`/`ProductVersion` собранного exe (ожидается `SWTools` / `1.0`).

## 3. Живой сквозной прогон — строго по runbook
Идти по `docs/release/BOEVOY_E2E_TEST_RUNBOOK_RU.md`, **Этапы 0–10**, критерии PASS/FAIL по
каждому окну — в `docs/audit/from-source-phase-3b-acceptance-TZ_RU.md` (сценарии **S1–S10**,
**L1–L5**). Чистый стенд лицензий — по процедуре из PR #51 (уже в main).

Краткая последовательность и **обязательные доказательства** по каждому шагу:

| Этап | Сценарий | Что сделать | Доказательство (приложить) |
|------|----------|-------------|----------------------------|
| 0 | — | зафиксировать сборку, код регистрации, донгл, сеть, серверный лог + слот БД «до» | вывод гейтов §2, версия exe |
| 1 | — | **бэкап** обеих веток реестра → **удалить целиком** `HKCU\SOFTWARE\SWTools` и `…\ZTool` → проверить отсутствие | скрины обоих `reg query` = «не удалось найти», `*.reg`-бэкапы |
| 2 | S9 | голая установка: exe + `SWToolsARM.dll` + каталоги «Шаблоны спецификации»/«Шаблон SW»/«Изображение»; **COM-зарегистрировать аддин** (`regasm SWTools.dll /codebase` + ключи `SOLIDWORKS\AddIns\{GUID}`, не просто копию рядом); запустить SW2025; **проверить, что аддин загрузился** — `GetAddInObject("SWTools.SwAddin")` даёт объект (не `Element not found/0x8002802B`), «Подключить SW» → `PartHits>0` | скрин дерева каталогов, лог загрузки аддина (объект получен), `PartHits>0` |
| 3 | S1,S2,S3,S4 | первый запуск без лицензии: заголовок «SWTools 1.0(x64)», UI русский без иероглифов; «О программе»; «Лицензия не обнаружена»; апдейт-чек инертен | скрины главного окна, About, License-not-found |
| 4 | L1 | на чистом стенде — должно быть «Лицензия не обнаружена» (а не молчаливый старт); на активированной — `verify_register` на сервер | серверный лог `verify_register`, скрин |
| 5 | L2 | онлайн-активация реальным кодом (Apply→Confirm); ключ в `HKCU\SOFTWARE\SWTools\sn` (**НЕ** под ZTool) | `reg query …\SWTools\sn`, лог/pcap, скрин успеха, слот «после» |
| 6 | L3 | перенос/снятие лицензии (Apply→Confirm); ключ очищается, слот освобождается | reg до/после, лог/pcap |
| 7 | L4 | негативные: неверный код, обрыв в середине, повторная активация занятого MNum | скрины русских ошибок, состояние сервера |
| 8 | L5 | сверка кадров протокола: тот же набор кадров/реакций на боевом exe и from-source | pcap/лог обоих, вывод сравнения |
| 9 | S7 | генерация спецификации/BOM в Excel-шаблон; нет китайского; Author=SWTools; русские `Col_*` без пустых строк; «Импорт…» на месте | экспортированный `.xlsx`, скрин «Сопоставление» и результата |
| 10 | S8 | прогнать Этапы 3–9 на боевом `SWTools.exe` и сравнить | заметки сравнения (единственное допустимое расхождение — пробелы в статус-строках) |

## 4. Формат отчёта (ОБЯЗАТЕЛЬНО сдать)
1. Создать `docs/audit/from-source-boevoy-acceptance-report_<YYYY-MM-DD>_RU.md` с:
   - таблицей **S1–S10 + L1–L5**: для каждого — `PASS`/`FAIL`/`N/A` + ссылка на файл-доказательство;
   - блоком «Гейты §2» с вердиктом по каждому;
   - окружением: версия exe (`ProductVersion`), версия SW, MNum, дата/время, ветка/коммит main.
2. Сложить все доказательства в `manual-test-reports/<YYYY-MM-DD>-boevoy-from-source/`
   (скрины, `*.reg` до/после, серверные логи/pcap, экспортированный `.xlsx`).
3. Вернуть в сессию краткую сводку: сколько PASS/FAIL, перечень FAIL, ссылки на отчёт и папку.

## 5. Эскалация
- Любой **FAIL по лицензии (L1–L5), BOM (S7) или сверке (S8/L5)** — остановиться и эскалировать
  с полным набором логов (reg до/после, серверный лог, pcap, скрин).
- Расхождение с боевым exe, кроме пробелов в статус-строках, считать FAIL и эскалировать.

## 6. Что НЕ делать
- Не тестировать на облачной VM (аппаратный гейт не пройдёт — результат недостоверен).
- Не запускать без предварительной полной очистки реестра обоих брендов (иначе L1/L2 невалидны).
- Не править исходники под прохождение теста; о любом баге — отчёт + эскалация.
