# Глубокий аудит лицензирования ZTool + отладка на боевом сервере

Дата: 2026-06-12
Сервер: `license.vizbuka.ru` / `185.112.102.122:58000`
Репозиторий: `lunin02118-ops/ztool` (ветка `main`, HEAD `1cb83bd`)

## 1. Итог (TL;DR)

- **Боевой сервер лицензий РАБОТАЕТ.** Я выполнил живую активацию прямо против
  `185.112.102.122:58000`: `apply_register(128)` → `result 13` + license blob
  (14808 байт), блоб проходит эмуляцию `SR.IsReg1`, machine-hash совпадает с
  ожидаемым. Полный TCP-протокол, крипто (RSA/AES), привязка к железу и ключ
  сервера — корректны.
- **Ключ клиента совпадает с ключом сервера.** Встроенный в перевыпущенный
  клиент публичный ключ (`client-rekey/new_key.txt`) **байт-в-байт совпадает** с
  `keys/public_key.txt` на боевом сервере. Клиент перенацелен на
  `185.112.102.122:58000`, версия `1.0` (getver = MD5("1.0 (x64)")). То есть
  клиент и сервер согласованы.
- **Корневая причина сбоя «получение лицензии»:** регистрационный код, который
  вводят в клиенте (`NMTPU45D-EZZ4F-4QHCC-9VKQP-UEYDBYUYJ`), **отсутствует в базе
  сервера**, поэтому сервер штатно отклоняет его кодом `无效注册码` (неверный код).
  Это видно в `audit_log` (записи 80–84 от 2026-06-12). Лечится добавлением
  нужного кода в БД (`cli add-code`). См. раздел 4 — нужно решение пользователя.

## 3. Что я сделал на боевом сервере

1. Сделал бэкапы `server.py` и `licenses.db` (`*.bak.<ts>`).
2. Выложил актуальный `server.py` из репозитория (hash совпал с `main`):
   - фикс обхода пароля (пустой пароль на защищённом коде теперь отклоняется);
   - `main()` теперь читает конфиг из env (`ServerConfig.from_env()`).
3. Переключил логирование с DEBUG на INFO (drop-in), `daemon-reload` + `restart`.
4. **Перепроверил вживую после деплоя:**
   - известный код + валидный fingerprint → `result 13` + валидный блоб (IsReg1 OK);
   - защищённый код + **пустой** пароль → `result 18` (неверный пароль) — отклонён;
   - защищённый код + **верный** пароль → `result 13` — принят.
5. Все временные тестовые коды/активации удалил — БД вернулась к исходным 4 кодам.

## 6. ГЛАВНЫЙ БАГ: онлайн-подтверждение (`register_confirm`) отклонялось

Найден в боевых логах: реальный клиент с машины `03000200-…` сегодня
**успешно прошёл `apply_register`** (id 90/91, `accepted`), но сервер
**отклонил `register_confirm(131)`** (id 92, 06:23:30):

```
register confirm: expected 4 branches + 2 times, got 5 fields
```

→ клиент показывает «Регистрация не удалась». Это и был реальный блокер
«получения лицензии» (apply проходил, а финальное подтверждение — нет).

### Корневая причина

`SR.get_rginfo()` собирает тело подтверждения через `StringBuilder.AppendLine`
из 7 элементов: 4 ветки + `RSA(время создания ключа HTwk2RCBDL)` +
`RSA(время создания 98CqyvBZcg)` + `RSA(DateTime.Now)`. На реальной сборке
клиента `GetRegistrycreatedtime.getregistryKeytime()` падает с
`Method not found: 'Int32 ZTool.GetRegistrycreatedtime.RegCloseKey(UIntPtr)'`
и возвращает `""`, а `RSAHelper.EncryptString("") == ""` — поэтому две строки
времени **пустые, но присутствуют** (AppendLine их всё равно выводит).

Серверный обработчик делал `fields = [ln for ln in lines if ln]` — то есть
**выкидывал пустые строки** до нарезки. Это сдвигало `RSA(now)` в слот времени,
давало 5 полей вместо 7 и возвращало `INFO_ERROR(6)`.

### Фикс (сервер, PR #9)

Парсинг теперь **позиционный**: внутренние пустые строки сохраняются (срезается
только хвостовой перевод строки), требуются 4 непустые ветки, а пустые/
отсутствующие поля времени трактуются как пустые → `b0/b1` шифруются ключом
`GD51("")`. `IsReg2` на той же сборке клиента тоже читает `""` → совпадает,
поэтому онлайн-подтверждение проходит **без перевыпуска клиента**.

Дополнительно: `writer.wait_closed()` обёрнут — клиент рвёт сокет (RST) сразу
после ответа, и это больше не логируется как unhandled-исключение asyncio.

### Проверка вживую против боевого сервера (после деплоя)

Прогон всех режимов (`tools/live_verify.py`, временный код + throwaway-машина):

```
[128 apply_register]   result=13  (IsReg1 OK)
[131 register_confirm] result=12  (IsReg2 OK)   [пустые времена — как у реального клиента]
[129 apply_remove]     result=11
[132 remove_confirm]   result=7
```

Тесты: `68 passed, 1 skipped` (добавлен `test_register_confirm_with_empty_creation_times`).

## 7. ВТОРОЙ БАГ: перенос лицензии (`apply_remove`) не возвращал blob

После фикса `register_confirm` пользователь перепроверил перенос: запрос уже
**доходит до сервера** (`apply_remove success`, `is_active=0`), но клиент всё
равно показывает «Не удалось перенести лицензию», локальная лицензия не снята,
а `remove_confirm(132)` не уходит.

### Корневая причина

По дизассемблеру (`TCPClient.SocketRecive` + `SR.outrg`): на ответ `129`
(result 11) клиент вызывает `SR.outrg(receive)` — **байт-в-байт как `SR.rg()`**:
расшифровывает тело AES-ключом `getver_today`, режет по `\t` и **требует ровно
4 поля** (4 ветки реестра). Только если `outrg` вернул `true`, клиент пишет
ветки и шлёт `verify_Remove(132)` с `get_rginfo()`. Наш сервер на `129`
отдавал **голый результат без тела** → `outrg` возвращал `false` →
«Не удалось перенести лицензию», подтверждение не отправлялось. Плюс слот
освобождался преждевременно → рассинхрон (сервер снял, клиент нет).

### Фикс (сервер, PR #9)

- `apply_remove(129)` теперь возвращает **result 11 + 4-веточный transport-blob**
  (`build_transfer_out_blob`), который `outrg` принимает (4 поля).
- Blob строится для зарезервированного «sentinel»-отпечатка
  (`TRANSFER_OUT_SENTINEL_MACHINE`), поэтому, переписав ветки, исходная машина
  **локально теряет лицензию** (её `IsReg1/IsReg2` больше не совпадает) — это и
  есть смысл «возврата лицензии на сервер» (floating). Слот при этом освобождается.
- Регресс-тест `test_transfer_out_flow` усилен: проверяет наличие blob, ровно 4
  поля и что bound-value отличается от активационного.

### Проверка вживую (после деплоя)

```
[129 apply_remove]     result=11   blob=14808B   fields=4   (outrg → true)
[132 remove_confirm]   result=7    («Лицензия успешно перенесена»)
```

## 8. Клиентские проблемы (вне сервера)

- **UI-дефект**: последнее из пяти полей кода в `FrmRg` слишком узкое (80px),

  клиента. Серверный фикс делает подтверждение рабочим независимо от этого.
