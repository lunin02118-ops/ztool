# Глубокий аудит лицензирования ZTool + отладка на боевом сервере

Дата: 2026-06-12
Сервер: `license.vizbuka.ru` / `185.112.102.122:58000`
Репозиторий: `lunin02118-ops/ztool` (ветка `main`, HEAD `1cb83bd`)

## 1. Итог (TL;DR)

- **Боевой сервер лицензий РАБОТАЕТ.** Я выполнил живую активацию прямо против
  `185.112.102.122:58000`: `apply_register(128)` → `result 13` + license blob
  (14808 байт), блоб проходит эмуляцию `SR.IsReg1`, machine-hash совпадает с
  ожидаемым. Полный TCP-протокол, крипто (RSA/AES), привязка к железу и ключ
  сервера — корректны.
- **Ключ клиента совпадает с ключом сервера.** Встроенный в перевыпущенный
  клиент публичный ключ (`client-rekey/new_key.txt`) **байт-в-байт совпадает** с
  `keys/public_key.txt` на боевом сервере. Клиент перенацелен на
  `185.112.102.122:58000`, версия `1.0` (getver = MD5("1.0 (x64)")). То есть
  клиент и сервер согласованы.
- **Корневая причина сбоя «получение лицензии»:** регистрационный код, который
  вводят в клиенте (`NMTPU45D-EZZ4F-4QHCC-9VKQP-UEYDBYUYJ`), **отсутствует в базе
  сервера**, поэтому сервер штатно отклоняет его кодом `无效注册码` (неверный код).
  Это видно в `audit_log` (записи 80–84 от 2026-06-12). Лечится добавлением
  нужного кода в БД (`cli add-code`). См. раздел 4 — нужно решение пользователя.

## 2. Что было на боевом сервере (до моих правок)

- systemd-сервис `ztool-tcp-server.service` — `active (running)`, слушает
  `0.0.0.0:58000`, БД `/opt/ztool-tcp-server/licenses.db`.
- Развёрнутый код **отставал** от репозитория:
  - `server.py` — старее `main`, **без фикса обхода пароля** (коммит `45b54e2`):
    строка `if password and not check_password(...)` пропускала проверку при
    пустом пароле → защищённый код (в т.ч. owner-код `57624D05-…`,
    пароль `zDjdf021189`) можно было активировать с **пустым** паролем. Дыра.
  - `main()` игнорировал env `ZTOOL_KEYS_DIR/ZTOOL_DB_PATH` (работало лишь по
    совпадению значений с дефолтами) — хрупко.
  - `machineid.py` отличался только переводами строк (CRLF), функционально идентичен.
- На сервисе было включено **DEBUG-логирование** (drop-in `debug.conf`), которое
  пишет расшифрованные тела запросов в journal — нежелательно для production.

## 3. Что я сделал на боевом сервере

1. Сделал бэкапы `server.py` и `licenses.db` (`*.bak.<ts>`).
2. Выложил актуальный `server.py` из репозитория (hash совпал с `main`):
   - фикс обхода пароля (пустой пароль на защищённом коде теперь отклоняется);
   - `main()` теперь читает конфиг из env (`ServerConfig.from_env()`).
3. Переключил логирование с DEBUG на INFO (drop-in), `daemon-reload` + `restart`.
4. **Перепроверил вживую после деплоя:**
   - известный код + валидный fingerprint → `result 13` + валидный блоб (IsReg1 OK);
   - защищённый код + **пустой** пароль → `result 18` (неверный пароль) — отклонён;
   - защищённый код + **верный** пароль → `result 13` — принят.
5. Все временные тестовые коды/активации удалил — БД вернулась к исходным 4 кодам.

## 4. Что требует решения пользователя — провижининг кода

Реальная машина (UUID `03000200-0400-0500-0006-000700080009`, диск
`YS202108310001100161`) уже имеет активную лицензию на **тестовом** коде
`ABCDEFGH-IJKLM-NOPQR-STUVW-XYZABCDEF` (артефакт прошлых тестов Devin).

Коды в БД сейчас:
- `57624D05-A7674-52386-5F315-A884DF6CC` — «owner primary key», пароль `zDjdf021189`, лимит 1, **свободен**.
- `ABCDE-FGHIJ-KLMNO-PQRST-UVWXY` — пароль `testpw` (тест).
- `ABCDEFGH-IJKLM-NOPQR-STUVW-XYZABCDEF` — без пароля (тест), занят реальной машиной.
- `DEADBEEF-A1B2C-3D4E5-6F7A8-9B0C1D2E3` — без пароля (тест), занят sim-машиной.

Кода `NMTPU45D-EZZ4F-4QHCC-9VKQP-UEYDBYUYJ` в БД нет — поэтому активация падала.
Нужно выбрать, какой код выдать на реальную машину (см. вопрос в чате).

## 5. Прочие находки (репозиторий)

- **Баг в `db.py` (исправлен, PR #9):** `os.makedirs(os.path.dirname(db_path))`
  падал на «голом» относительном пути (`--db licenses.db` → `os.makedirs('')` →
  `FileNotFoundError`). Я это поймал вживую при провижининге. Фикс + тест,
  локально `67 passed, 1 skipped`. PR: https://github.com/lunin02118-ops/ztool/pull/9
- `register_confirm (131)` по-прежнему не связывает confirm с конкретным
  apply/кодом (совместимо с клиентом, но стоит ужесточить для production).
- RSA без паддинга / AES IV=key — по необходимости совместимости с клиентом, не меняется.
