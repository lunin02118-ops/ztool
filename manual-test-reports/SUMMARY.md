# Manual test summary (sanitized)

Подробные отчёты (скриншоты, локальные пути, серверные детали) намеренно
**не** хранятся в репозитории. Ниже — краткая сводка проверок по этой ветке.

## Лицензирование (серверная часть)

Перевыпущенный клиент `ZTool.exe` проверен против боевого license-сервера —
все 4 режима протокола проходят:

| Режим | Запрос | Результат |
| --- | --- | --- |
| Регистрация | `apply_register` (128) | `13` + license blob |
| Подтверждение | `register_confirm` (131) | `12` (IsReg2 OK) |
| Перенос (отвязка) | `apply_remove` (129) | `11` + transfer blob |
| Подтверждение переноса | `remove_confirm` (132) | `7` (слот освобождён) |

TCP-сервер переведён на единую БД с админкой: после активации в админке
видны привязка машины и счётчики; после переноса — очищаются.

## BOM-экспорт

- Шаблон открывается по абсолютному пути `<bomname>` (относительный путь
  ломается, т.к. резолвится относительно `…\Documents` под SolidWorks).
- Открытие `.xlsx` через NPOI требует `ICSharpCode.SharpZipLib.dll` 0.86.0.518
  (добавлен в поставку).
- Имена колонок сопоставляются с именованными диапазонами шаблона через
  `namemappinglist`.
- Live-прогон на SolidWorks 2025 зафиксирован в
  `manual-test-reports/PR8_BOM_LIVE_TEST_20260612.md`.
  Первичные прогоны показали: чтение SW работает (`29 поз.`), но при
  укороченных demo-cn `propname` файлы `.xlsx` содержали только шаблон без
  строк данных.
- Повторный прогон с `client-core/tools/ZTool.demo-cn.settings` на китайской
  demo-модели также дал пустой Excel (`nonempty_after_row6 = 0`), поэтому
  причина не сводится к русским именам свойств в профиле.
- Скрин свойств детали `0614-P001` показал реальные полные имена свойств:
  `设计`, `零件名称`, `图号`, `材料`, `类型`, `版本`, `表面处理`, `设计日期`,
  `重量`. Укороченные `设/零名/图/材/...` не совпадали с моделью.
- Повторный прогон 2026-06-13 с `ZTool.demo-cn-fixed.settings` подтвердил:
  пустой экспорт исправлен. Файл
  `D:\ztool-pr8-test\bom-exports\0614-A00-20260613-0019.xlsx` содержит
  32 строки данных, заполнены `Наименование`, `Обозначение`, `Версия`,
  `Тип обработки`, `Обработка поверхности`, `Масса`, `Материал`.
- Остаточный статус BOM: `PARTIAL PASS`. Данные custom-свойств экспортируются,
  но служебные колонки `№ п/п` и `Кол-во` остались пустыми; также один раз
  зафиксирован диалог `RPC_E_CANTCALLOUT_ININPUTSYNCCALL` после подключения SW.
- Повторный прогон commit `7983769` проверил патч `Количество`: pre-flight
  показывает `Col_Number`/`Col_Quantity` как `anchor OK`, и в таблице ZTool
  `Номер`/`Количество` заполнены. Первичный файл
  `0614-A00-20260613-0020.xlsx` был противоречивым, но свежие экспорты
  `0614-A00-20260613-0927.xlsx` и `0614-A00-20260613-0928.xlsx` подтвердили:
  `№ п/п`, `Кол-во` и `Путь` записываются в Excel. Служебные колонки по
  commit `7983769` считаются `PASS`.
- В этих свежих файлах custom-колонки пустые, потому что runtime-профиль
  из `ZTool.settings` русский (`Имя детали`, `Номер чертежа`, `Материал`),
  а demo-модель китайская (`零件名称`, `图号`, `材料`). Для полного demo-green
  нужен прогон `7983769` с demo-cn-fixed профилем.
- Commit `c633622` добавил 8 русских режимов экспорта и `FilterRulesList`.
  Runtime-проверка: `check_bom_template.py` = `PASS`, в меню
  `Экспорт спецификации` отображаются все 8 режимов, включая
  `Обрабатываемые детали` и `Покупные изделия`. Фильтр-режимы требуют
  отдельной проверки на моделях с реальными значениями свойства `Тип`, т.к.
  текущие фильтры используют заглушки.
- Боевой прогон commit `ca1e8b3` выполнен на реально открытой сборке
  `0614-A00.SLDASM`: русские свойства записаны в модель через SolidWorks COM,
  файлы сохранены, после `Подключить SW` таблица ZTool заполнена 29 строками.
  Все 8 режимов создали `.xlsx` в `D:\ztool-pr8-test\bom-exports\full-test`.
- Режимы 1-6 прошли: сводная 29 строк, иерархия 32 строки, верхний уровень
  6 строк, детали 25 строк; `№ п/п`, `Кол-во`, `Путь` и custom-свойства
  записаны; режимы с эскизами содержат 29/32 изображения.
- Режимы 7-8 (`Обрабатываемые детали`, `Покупные изделия`) не прошли:
  экспортируют полный список 29 строк. Диагностика с точными значениями
  фильтра `机加`/`外购` и с переключением `ByRuler/ByFilter` также дала полный
  список, значит `RulesList -> FilterRulesList` фактически не применяется при
  BOM-export.
- `validate_bom_exports.py` сейчас даёт ложный `FAIL`, потому что считает
  форматированные пустые строки до `ws.max_row`. Ручная проверка `openpyxl`
  по заполненным ключевым ячейкам подтверждает PASS для режимов 1-6.
- Перепроверка commit `fc7d801`: новый `ZTool.settings` с TAB-разделителем
  и русскими операторами правил прошёл pre-flight `[1c]`. На той же открытой
  сборке режим 7 (`Обрабатываемые детали`) выгрузил 15 строк только с
  `Тип=机加`, режим 8 (`Покупные изделия`) выгрузил 9 строк только с
  `Тип=外购`. Обновлённый валидатор дал `PASS`. Дефект фильтрованных режимов
  7-8 закрыт.
- Перепроверка commit `dca7a84`: runtime-шаблон заменён, `G7.font=Arial`,
  лист `Спецификация`, печать `A1:O75`, `landscape`, `fitToWidth=1`.
  Live-экспорт сводной спецификации дал 29 строк; в диапазоне данных
  `A7:O35` все 435 ячеек имеют `Arial`, `宋体=0`, `№ п/п`/`Кол-во`/`Путь`
  заполнены, валидатор `PASS`. Остаточное замечание: `宋体` всё ещё есть в
  пустых ячейках шаблона/выходного файла (`B1:K3`, часть строк `72:75`), не
  влияющих на клонируемую строку данных, но это не "нулевой остаток" по всей
  книге.
- Перепроверка commit `82fe830`: шаблон в runtime заменён, raw-проверка
  `styles.xml` дала `CJK font names=0`, `charset="134"=0`, per-cell
  `Arial=1425`, `宋体=0`, pre-flight `PASS`. То есть полный purge шрифтов
  в самом шаблоне подтверждён. Live-export на этом коммите заблокирован:
  при `Подключить SW` на реально открытой `0614-A00.SLDASM` падает
  `System.NullReferenceException` в `ZTool.PMPHandler.DefWndProc(Message& m)`,
  таблица ZTool остаётся пустой (`0` строк). Отдельно зафиксировано, что
  включение version-specific автозагрузки add-in для `SOLIDWORKS 2025`
  воспроизводит стартовую ошибку `Не удалось загрузить Microsoft .NET Framework`;
  ключ возвращён в `0`.
