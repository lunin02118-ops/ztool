# ЧАСТЬ A. ПОЛНОЕ ЗАДАНИЕ

## A.0 Контекст
ZTool — коммерческий аддон к SolidWorks (китайского вендора), состоящий из трёх частей:
- **`ZTool.dll`** — COM-аддон SolidWorks (лента/PropertyManager, чтение сборок/BOM). Защищён протектором (тело методов и ресурсы расшифровываются только внутри живого процесса `SLDWORKS.exe`).
- **`ZTool.exe`** — внешний лаунчер («Управление файлами», пакетные операции, печать, объединение PDF, «О программе»). Запускается **самим аддоном** через `Process.Start` из папки, где лежит `ZTool.dll`.
- **Сервер лицензий** (Python) — выдача/проверка лицензий.

Цель проекта: интегрировать PR #1–#6 в одну ветку, локализовать BOM-шаблоны и **закрыть 3 дефекта** перед merge PR #7.

## A.2 Производные требования, добавленные по ходу (явные просьбы пользователя)
1. **Русификация `ZTool.exe`** — триал-таймер, статус-бар, заголовки таблиц, диалог ошибок (остаточный китайский после первой локализации).
2. **«Обновления вообще вырубить»** — полностью отключить сетевую проверку обновлений (запрос к `z-tool.cn`) и попап «Найдена новая версия». У пользователя в планах свой сервер обновлений.
3. **«Иконки сделай»** — восстановить иконки ленты (лежат в рантайм-сателлите протектора) из полного дампа памяти процесса SW и вшить в DLL.
4. **Регрессия чтения** — после моей локализации `ZTool.exe` чтение из SW стало возвращать **0 позиций** («Подключение завершено», но таблица пустая). Найти причину и починить.
5. **Методика тестирования всего приложения** — единый прогон DLL+EXE+сервер.

## A.3 Найденная первопричина регрессии чтения (ключевой технический результат)
Чтение сборки идёт по IPC между `ZTool.exe` и `ZTool.dll`:
- exe отправляет аддону пакет `"ZToolRequest@001" + code::Getpkt()` (через `WM_COPYDATA`/`SendMessage`);
- `Getpkt()` вычисляет токен из `GetEntryAssembly().GetName().GetPublicKeyToken()` (**public key самого exe**);
- аддон в `PMPHandler::DefWndProc` сверяет пакет с ожидаемым токеном **оригинального** exe (`f08fc7047657204e`, зашит на стороне DLL).

Мой инструмент локализации (`Localizer`) при пересборке exe **срезал strong-name public key** (`asm.PublicKey = null`), чтобы exe грузился на машинах с проверкой подписи. Из-за этого `GetPublicKeyToken()` возвращал пусто → `Getpkt()` давал неверный токен → аддон **молча отклонял** запрос → данные не приходили → **0 позиций**.

**Фикс:** сохранить public key (токен снова `f08fc7047657204e`), сняв только COR20-флаг `StrongNameSigned` (delay-signed). Стартовый exe (запуск как процесс) strong-name не проверяет, поэтому грузится везде, а токен рукопожатия снова совпадает. Чтение в `GetDataByBom` лицензией **не гейтится** (проверено) — «лицензия не обнаружена» лишь косметический статус.

## A.4 Границы скоупа
**Входит:**
- Текстовая русификация UI (DLL и EXE) без изменения логики/сигнатур.
- Восстановление иконок из дампа (байт-в-байт оригинал).
- Отключение апдейт-чека.
- Фикс пайплайна сборки exe (сохранение public key).
- Фиксы сервера (установка, env-переменные).

**НЕ входит / запрещено:**
- Изменять бизнес-логику, структуру классов, сигнатуры методов DLL/EXE.
- Восстанавливать/ломать защиту сверх необходимого для русификации.
- Тестировать DLL/EXE в SolidWorks на моей стороне (SW COM виснет на VM — финальную боевую проверку чтения делает пользователь).
- Заявлять успех без положительного отчёта пользователя.
- force-push в main/master, коммит секретов, sudo для git.

# ЧАСТЬ B. МЕТОДИКА АУДИТА И ПРОВЕРКИ РЕАЛИЗАЦИИ

Семь уровней — от кода к бою. Каждый уровень имеет цель, шаги, инструменты и критерий прохождения. Принцип: **дефект ловится на максимально раннем уровне**; боевой тест — финальное подтверждение, а не первая линия.

## Уровень 0. Целостность артефактов и происхождение
**Цель:** убедиться, что тестируется именно тот файл, а не старая копия (главная причина ложных «провалов» в этом проекте).
- Сверка SHA256 каждого артефакта с таблицей A.6:
  ```powershell
  Get-FileHash <файл> -Algorithm SHA256
  ```
- Зафиксировать соответствие «коммит ↔ хеш ↔ путь в репо».
- **Критерий:** хеши совпадают; в рабочей папке нет посторонних `ZTool.exe/ZTool.dll`.

## Уровень 1. Статический анализ исходного кода (то, что в репо как текст)
**Цель:** проверить правки, которые живут в исходниках (сервер Python, `Localizer`, `build.ps1`, BOM-шаблоны).
- **Сервер (Python):**
  - `ruff`/`flake8` + `mypy` (если настроены), ревью diff по установке и чтению `ZTOOL_HOST/PORT/KEYS_DIR/DB_PATH`.
  - Проверить, что значения по умолчанию не перетирают env, и нет хардкода путей.
- **Localizer/build.ps1:**
  - Ревью `StripStrongName` — что public key **сохраняется**, снимается только флаг `StrongNameSigned`.
  - Ревью таблицы переводов: ключи протокола/конфигов/имена контролов/шрифты/пути help **не** переведены (только видимый текст).
- **Критерий:** линтеры зелёные; в diff нет изменения логики/сигнатур; «опасные» строки в стоп-листе перевода.

## Уровень 2. Бинарный / IL-аудит собранных DLL и EXE
**Цель:** проверить то, что нельзя увидеть в исходниках (DLL защищена; exe собирается реинжектом). Инструменты — мои утилиты в `C:\Users\Administrator\inspect\` (dnlib): `Il.exe`, `StrFind.exe`, `AsmId.exe`, `Cor20.exe`, `FindMeth.exe`.
1. **Идентичность сборки и токен:**
   ```
   AsmId.exe ZTool.exe   → PKToken = f08fc7047657204e (НЕ "(none)")
   Cor20.exe ZTool.exe   → ILOnly, StrongNameSigned=False, HasPublicKey=True
   ```
2. **Полнота локализации:** скан остаточного китайского (CJK) во всех `ldstr` и `.resources`; сверка со стоп-листом «намеренно не переведено».
3. **Отключение апдейта:** `Il.exe ZTool.exe CheckUpdate getinfo` → пустой `ret`; `StrFind.exe ZTool.exe "z-tool"` и `"3.8.5"` → пусто; точки показа окна нейтрализованы.
4. **IPC-контракт:** в DLL `StrFind.exe … "ZToolRequest@001"` → обработчик `PMPHandler::DefWndProc`; путь чтения `GetDataByBom` не содержит license/trial-гейтов.
5. **Иконки:** все ресурсы ленты присутствуют в DLL и декодируются GDI (валидные BMP/PNG-страйпы); зависимость от сателлита убрана; GUID аддона сохранён.
6. **Загружаемость:** `Assembly.LoadFile/LoadFrom` для DLL → OK; для exe помнить: как **библиотека** delay-signed даёт `0x8013141A` — это норма, проверять надо запуск как **процесс**.
- **Критерий:** токен на месте, остаточного видимого CJK нет, апдейт заглушён, IPC-обработчик цел, иконки валидны.

## Уровень 3. Проверка пайплайна сборки (воспроизводимость)
**Цель:** сборка детерминирована и не вносит регрессий.
- Чистый прогон `client-core/build.ps1` (нужен `dotnet` в PATH, у меня `C:\dotnet`).
- Контроль вывода: `dangling typerefs = 0`, шаг verify проходит, `strongname-stripped=0`.
- Хеш выходного `out/ZTool.exe` совпадает с эталоном A.6.
- **Критерий:** сборка из чистого состояния даёт эталонный бинарь без висячих ссылок.

- **Критерий:** `pytest` зелёный (эталон: 66 passed, 1 skipped); сервер поднимается по заданным env.

## Уровень 5. Развёртывание на стенде (smoke перед боем)
**Цель:** собрать рабочую раскладку, как у конечного пользователя.
- ASCII-путь без пробелов/кириллицы (напр. `D:\ztool-test\`), удалить ВСЕ старые копии.
- В одной папке: `ZTool.dll` + `ZTool.exe` + зависимые DLL (`Ribbon`, `ExpandableGridView`, `NPOI`, `itextsharp`, …) + `SolidWorksTools.dll`.
- Регистрация аддона строго из этой папки:
  ```
  RegAsm /codebase ZTool.dll
  ```
  затем проверить, что `CodeBase` в реестре указывает **именно сюда** (а не на старый путь).
- **Критерий:** `Assembly.LoadFile(ZTool.dll)` = OK; реестр аддона указывает на нужную папку; рядом лежит правильный `ZTool.exe` (хеш `f6822e6c…`).

> ⚠️ Ключевой риск проекта: аддон запускает `ZTool.exe` через `Process.Start` из **папки своего `ZTool.dll`** (`SwAddin::openZtool`). Если SW грузит DLL из другого зарегистрированного пути — будет запущен **старый** exe оттуда, а не ваш. Поэтому новый exe кладём в ту же папку, что и загруженный SW `ZTool.dll`.

## Уровень 6. Боевые тесты на реальной машине с SolidWorks 2025
**Цель:** финальное подтверждение поведения в живом SW (только на машине пользователя — у меня SW COM виснет).
1. **Загрузка аддона:** Tools → Add-Ins, без ошибок старта; вкладка ZTool видна.
2. **Лента:** все кнопки/тултипы на русском; **иконки на месте** (скрин).
3. **In-DLL кнопки** (без exe): «Переименование деталей», «Сохранить выбранное», «Случайная окраска» и т.д. — панель PropertyManager на русском, не падает.
4. **EXE-лаунчеры:** «Управление файлами», «Пакетная печать», «О программе» — окна открываются; триал/статус/таблицы/диалог ошибок на русском; **апдейт-попап НЕ всплывает**.
5. **Главный тест — чтение из SW** (закрывает регрессию):
   - открыть сборку (напр. `0614-A00.SLDASM`), «Управление файлами» → «Подключить SW» → читать;
   - проверить **оба режима**: «по спецификации» и «читать всё»;
   - пока окно открыто, зафиксировать, какой exe реально запущен:
     ```powershell
     Get-Process ZTool | Select-Object Id, Path
     Get-FileHash "<путь оттуда>" -Algorithm SHA256   # ожидаем f6822e6c…
     ```
   - **Критерий:** таблица заполняется **N>0 позиций** (а не 0).
6. **Независимый smoke модели/SW** (отделить дефект приложения от среды): через SW API записать и прочитать произвольное свойство (напр. `ZTOOL_CODEX_SMOKE=OK`, `Save=True`) — доказывает, что SW и модель исправны, и дефект (если есть) в логике приложения.
7. **Сервер в связке:** клиент ходит на свой сервер лицензий по `ZTOOL_*`; активация/триал отрабатывают.
- **Критерий уровня:** все пункты «зелёные», главное — чтение даёт позиции.
